# 0518 联机规划：桃源乡如果真要做联机，它可以长成什么样

## 0. 文档定位

这不是一份“加个多人模式”的文档。

它也不是一份“我们谨慎一点，先做一点点社交功能”的文档。

这份文档只回答一个问题：

> 如果不考虑工作量，不考虑工期，不考虑技术债规模，《桃源乡独立版》最值得想象成怎样的联机世界？

所以这版会更激进一些，更像一张总蓝图，而不是执行拆单。

## 1. 我对桃源乡联机的判断

### 1.1 这游戏不该变成标准网游

《桃源乡》的核心不是战斗排位，也不是跑图打本，而是：

- 经营节奏
- 日历驱动
- 资源规划
- 生活感
- 关系感
- 节气与节庆
- 村庄与庄园的成长痕迹

所以它不适合照搬：

- MMO 的刷怪逻辑
- 竞技网游的段位逻辑
- 自由市场的高频交易逻辑
- 大世界实时抢点逻辑

这些东西一旦塞进来，气质会变。

### 1.2 但它非常适合做“有社会关系的经营世界”

如果换一个方向看，《桃源乡》其实天生适合联机，只是它更适合联成这种形态：

- 邻里互助
- 村社共建
- 来访参观
- 书信往来
- 节会同游
- 协作委托
- 共享庆典
- 互相成就

也就是说，联机不是拿来替代经营，而是把经营放进一个更有温度的社会网络里。

### 1.3 这款游戏最强的联机价值，不是同屏，而是关系网

真正能让这游戏一旦联机就“活起来”的，不是看见别人的角色在田里跑，而是：

- 我知道谁今天在忙什么
- 我能去别人家看看他最近怎么装饰的
- 我能帮他补一单稀缺材料
- 我能把今天钓到的特殊鱼寄给他
- 我能和他一起完成周目标
- 我能在节庆里和别人形成临时队伍

这才是桃源乡联机最该长出来的东西。

## 2. 终局愿景：桃源乡联机宇宙

如果不考虑工作量，我会把桃源乡的联机最终目标定成一个很大的东西：

> 一个以庄园为单位、以村社为关系网、以节气为节奏、以协作为主要互动方式的生活型在线世界。

### 2.1 世界不是一张图，而是一张社会地图

玩家不只是有一个角色，而是有：

- 一个庄园
- 一个村社身份
- 一组好友和邻里
- 一份公开名片
- 一串来访记录
- 一套对外展示的经营风格

这意味着玩家的“世界坐标”不只是地理坐标，而是社会坐标。

### 2.2 每个玩家都该有一个“可被看见的经营人格”

联机最重要的是让每个玩家的风格被别人看出来。

比如：

- 这个人擅长种植
- 这个人鱼塘很强
- 这个人博物馆收藏特别全
- 这个人很会搞节庆
- 这个人常常接委托
- 这个人庄园布置非常有审美
- 这个人是村社里的材料供应者

联机不是把玩家变成同质化数值单位，而是让每个人长出自己的社会标签。

### 2.3 联机世界的核心循环

我会把联机世界的循环定义成这样：

1. 我经营自己的庄园。
2. 我的经营结果被别人看见。
3. 别人的需求又反过来影响我的经营优先级。
4. 节庆、委托、共同目标把大家重新拉回同一个节奏。
5. 世界因为这些往来而持续变化。

这不是一个“多人在线游戏”的循环，这是一个“互相牵连的生活世界”的循环。

## 3. 如果不计成本，桃源乡可以做成哪些联机层

我会把联机拆成五层宇宙，从轻到重一直往上叠。

### 3.1 第一层：可见联机

这一层的目标是：让别人存在于你的世界里。

内容可以包括：

- 好友系统
- 邻里系统
- 玩家名片
- 公开庄园页
- 最近来访记录
- 点赞、收藏、留言
- 一键关注某种经营风格
- 热门庄园榜
- 本周高人气庄园展

这一层的意义是“看见”。

### 3.2 第二层：交换联机

这一层的目标是：让玩家之间发生资源和情感的交换。

内容可以包括：

- 送礼
- 寄信
- 借种
- 借鱼苗
- 借展品
- 材料求助
- 订单转接
- 代收代交
- 限量交换站
- 周期性集市

这一层的意义是“往来”。

### 3.3 第三层：协作联机

这一层的目标是：让玩家共同完成一件事。

内容可以包括：

- 协作委托
- 村社任务
- 共同修复工程
- 节庆筹备
- 限时采集
- 组合钓鱼节
- 双人农忙
- 协作矿洞
- 搭档护送
- 合作经营订单

这一层的意义是“共同推进”。

### 3.4 第四层：同场联机

这一层开始进入实时互动。

内容可以包括：

- 节庆现场
- 集市现场
- 共同守夜
- 共同赛舟
- 共同拼菜
- 共同答题
- 共同造景
- 一起跑短时活动房间

这一层的意义是“同场体验”。

### 3.5 第五层：共治联机

这一层是终局层。

玩家不只是一起玩，而是一起塑造世界。

内容可以包括：

- 村社议事
- 公共建设投票
- 社区目标决策
- 节气活动策划
- 资源分配提案
- 新设施落地投票
- 世界事件响应
- 共识型长线项目

这一层的意义是“共同治理”。

## 5. 我会把“联机”做成怎样的结构

如果不考虑工作量，我会直接把桃源乡的联机结构做成四块：

### 5.1 个人世界

每个玩家保留自己的单人经营核心。

- 农场
- 背包
- 时间
- 探索
- 剧情
- 养成
- 收藏

这块不消失，反而是所有联机关系的根。

### 5.2 社会世界

这是新增的在线关系层。

- 好友
- 邻里
- 村社
- 师徒
- 访客
- 关系标签
- 行为记录
- 风格档案

这块负责让玩家彼此连起来。

### 5.3 共享世界

这是公共层。

- 公共事件
- 公共节庆
- 公共建设
- 公共榜单
- 公共集市
- 公共目标

这块负责让大家共享同一节奏。

### 5.4 活动世界

这是房间层。

- 节庆房间
- 协作房间
- 比赛房间
- 探索房间
- 特殊事件房间

这块负责把多人体验做得像样。

## 6. 激进技术蓝图：如果真往大了做，底层要怎么分

不考虑工作量的话，我不会只在现有 REST API 旁边加几个接口。

我会把联机底层拆成一套独立的“桃源在线运行层”。

### 6.1 核心原则：个人存档不是多人共享状态

哪怕做得再激进，我也不会让多人直接共同编辑同一个完整存档。

原因很简单：

- 当前单人存档很厚
- 经营状态跨 store 很多
- 多人同时改一个大 JSON，冲突会非常难处理
- 玩家离线、断线、切设备后很容易覆盖别人结果

所以更合理的结构是：

- 个人存档：保留个人世界
- 联机事件：记录跨玩家互动
- 联机房间：承载实时活动
- 联机结算：把奖励安全写回个人存档

一句话：

> 多人互动发生在联机层，个人成长落回个人存档。

### 6.2 服务端权威应该管什么

服务端必须权威的内容：

- 玩家身份
- 好友关系
- 邻里关系
- 村社成员
- 公共榜单
- 公共活动进度
- 求助单状态
- 交易与赠礼流水
- 房间状态
- 活动结算
- 惩罚、补偿、审计日志

客户端可以保留权威的内容：

- 单人农场即时操作
- 单人 UI 状态
- 本地演出
- 非联机剧情推进
- 离线模式临时状态

客户端可以预演、服务端确认的内容：

- 送礼
- 接单
- 交付
- 点赞
- 留言
- Ready
- 房间小动作
- 活动选择

### 6.3 桃源在线运行层建议模块

如果完整建设，我会拆这些服务端模块：

- `taoyuanIdentityRuntime`
- `taoyuanSocialRuntime`
- `taoyuanManorRuntime`
- `taoyuanVisitRuntime`
- `taoyuanCoopOrderRuntime`
- `taoyuanGiftRuntime`
- `taoyuanNeighborRuntime`
- `taoyuanSocietyRuntime`
- `taoyuanFestivalRuntime`
- `taoyuanRankingRuntime`
- `taoyuanMarketRuntime`
- `taoyuanActivityRoomRuntime`
- `taoyuanWorldEventRuntime`
- `taoyuanAuditRuntime`
- `taoyuanModerationRuntime`

当前执行命名已落到 `docs/online/06-naming.md`，后续 runtime、store 与事件名统一按那份规范扩展，不再临时散写。

它们的分工大概是：

- 身份层管账号与公开资料
- 社交层管好友、关注、黑名单
- 庄园层管公开快照和展示配置
- 来访层管访问记录、留言、点赞
- 委托层管求助单、协作单、状态流转
- 礼物层管寄送、领取、补偿、限制
- 邻里层管小组关系
- 村社层管组织、职位、公共目标
- 节庆层管周期活动和主题活动
- 榜单层管统计快照
- 市集层管寄售、交换、限购
- 房间层管实时活动
- 世界事件层管全服或分区事件
- 审计层管所有跨玩家变更
- 审核层管文本、图片、举报和封禁

### 6.4 数据模型建议

最核心的数据对象应该有这些：

- `PlayerProfile`
- `PublicManorSnapshot`
- `VisitRecord`
- `GuestbookEntry`
- `FriendRelation`
- `NeighborGroup`
- `VillageSociety`
- `CoopOrder`
- `GiftParcel`
- `ExchangeListing`
- `FestivalCampaign`
- `RankingBoard`
- `WorldEvent`
- `ActivityRoom`
- `RoomAction`
- `SettlementReceipt`
- `AuditLog`

其中最重要的是 `SettlementReceipt`。

它要记录：

- 来源活动
- 来源玩家
- 目标玩家
- 目标存档槽
- 奖励明细
- 幂等 key
- 是否已写入
- 是否已领取
- 是否补偿过
- 创建时间
- 过期时间

只要跨玩家奖励能做到“可追踪、可补发、可撤销、可解释”，联机系统就会稳很多。

### 6.5 存储路线

如果不考虑工作量，我会分三档：

#### A. 早期原型

- JSON 文件
- 原子写
- 模块锁
- 少量接口

适合快速验证玩法。

#### B. 正式联机

- MySQL / PostgreSQL
- 关键表结构化
- 事务保护
- 审计表
- 排行榜快照表
- 房间运行时内存状态

适合公开运行。

#### C. 激进终局

- 关系型数据库存权威数据
- Redis 存房间、限流、在线状态
- 对象存储放庄园快照图、截图、附件
- 消息队列处理活动结算
- 定时任务生成榜单和节气事件

适合大规模在线运营。

### 6.6 实时通信路线

我会把实时通信拆成两层：

- 轻实时：SSE 或轮询，用于通知、榜单刷新、活动进度
- 强实时：WebSocket，用于房间玩法

轻实时适合：

- 好友上线提醒
- 来访提醒
- 求助单状态变化
- 邮件抵达
- 公共活动进度刷新

强实时适合：

- 房间 ready
- 同场活动动作
- 实时小游戏
- 协作探索
- 节庆现场

### 6.7 房间统一状态机

所有实时房间都用统一状态：

- `created`
- `inviting`
- `ready_check`
- `countdown`
- `running`
- `paused`
- `settling`
- `closed`
- `aborted`

房间事件统一：

- `room.join`
- `room.leave`
- `room.ready`
- `room.unready`
- `room.start`
- `room.action`
- `room.snapshot`
- `room.pause`
- `room.resume`
- `room.settle`
- `room.close`
- `room.reconnect`

这样后面不管是钓鱼赛、赛舟、矿洞、灯谜、守夜，都不用重做一套房间底层。

## 7. 激进玩法设计：各层到底怎么玩

### 7.1 公开庄园：每个玩家都是一个可参观地点

公开庄园不是截图页，而是玩家的社交门面。

它应该包含：

- 主视觉截图
- 当前季节
- 庄园主题
- 展示作物
- 展示鱼塘
- 展示宠物
- 展示收藏
- 展示称号
- 本周需求
- 可接求助
- 访客留言
- 主人的一句话

更激进一点，可以做：

- 庄园导览路线
- 访客自动路线回放
- 主题房间
- 庄园音乐
- 主人推荐参观点
- 公开摆摊位
- 本周庄园故事

### 7.2 访客系统：来访不是点一下，而是留下痕迹

来访可以拆成很多层：

- 看一眼
- 点赞
- 收藏
- 留言
- 送祝福
- 放一封信
- 放一个小礼物
- 接一个求助单
- 触发一次访客事件

可以有“访客足迹”：

- 今天谁来过
- 他看了哪里
- 他留下了什么
- 他帮了什么忙
- 他带走了什么委托

这会让玩家打开庄园时有一种很强的“有人来过”的感觉。

### 7.3 好友与邻里：不要只做列表，要做关系身份

好友不应该只是名字列表。

每个好友都可以有：

- 最近活跃
- 当前季节进度
- 主营方向
- 最近成就
- 可互助事项
- 最近求助
- 共同完成记录
- 关系称号

邻里则可以更进一步：

- 3-12 人小组
- 小组公告
- 共享任务板
- 共享捐献目标
- 邻里等级
- 邻里称号
- 邻里仓库
- 邻里节庆目标
- 邻里装饰物

### 7.4 村社：真正的多人组织

村社可以是桃源乡版公会，但不要叫公会味太重。

它可以叫：

- 村社
- 乡会
- 田社
- 山海会
- 同游社
- 百工社

村社可拥有：

- 社长
- 管事
- 匠人
- 采办
- 账房
- 记录人
- 普通成员

每个职位有轻权限：

- 审批加入
- 发起提案
- 发布任务
- 管理公共仓
- 设定本周目标
- 管理留言

### 7.5 协作委托：联机经营的核心发动机

协作委托应该成为第一批核心联机玩法。

委托类型可以有：

- 缺货求助
- 节庆备货
- 博物馆补展
- 鱼塘借种
- 育种认证
- 商会急单
- 村社建设
- 远征补给
- NPC 特殊请求
- 临时灾害应对

委托流程：

1. 玩家发布需求。
2. 系统校验需求是否合法。
3. 好友、邻里或公开玩家接单。
4. 接单者交付资源或完成任务。
5. 服务端生成结算凭证。
6. 双方获得奖励、声望或关系值。

更激进一点：

- 委托可以拆成多段
- 多人可以共同交付
- 委托有品质评价
- 委托会形成长期声誉
- 常帮某类忙的人会获得社会标签

### 7.6 送礼与书信：邮箱升级成情感通道

当前邮箱已经很适合继续扩展。

玩家之间可以寄：

- 普通信
- 节日贺信
- 小礼物
- 材料包
- 种子包
- 鱼苗
- 装饰物
- 纪念品
- 合照
- 活动邀请

可以做得更有味道：

- 春信
- 夏帖
- 秋笺
- 冬书
- 节气明信片
- 来访感谢信
- 师徒赠言

书信不是纯功能，它会给联机加很多温度。

### 7.7 集市与交换站：可以做，但必须慢

如果要做交易，我建议先做“慢交易”，不要做自由拍卖行。

更适合桃源乡的是：

- 每周交换站
- 限量寄售
- 邻里内部换物
- 指定品类交换
- 节庆物资互换
- 官方调控价格区间

不要一开始做：

- 实时拍卖
- 无限挂单
- 金币自由购买所有物资
- 高阶材料完全市场化

激进不等于放飞经济。

### 7.8 公共建设：玩家一起改变世界

这是非常值得做大的方向。

公共建设可以包括：

- 修桥
- 修码头
- 建集市
- 修灯会街
- 建温泉
- 开山路
- 扩仓库
- 修祠堂
- 建书院
- 开港口

建设不是单纯捐材料，而是：

- 玩家投票决定先建什么
- 玩家分工提交材料
- 村社贡献会被记录
- 完成后全服或分区世界变化
- 有贡献者纪念碑

这会让玩家感到世界真的被大家一起推动了。

### 7.9 节庆同场：联机最该闪光的地方

桃源乡最适合把实时联机放在节庆里。

比如：

- 元日守岁
- 上元灯会
- 清明踏青
- 端午赛舟
- 七夕同游
- 中秋赏月
- 重阳登高
- 腊八共煮
- 年末祭灶

每个节庆可以有：

- 公共场景
- 小队活动
- 限时任务
- 共同进度
- 节庆榜单
- 节庆纪念品
- 节庆合照

### 7.10 协作探索：让远征变成多人故事

协作探索可以先从短房间做起。

玩法候选：

- 双人矿洞
- 三人采集
- 四人护送
- 夜路守灯
- 山道修复
- 海域探宝
- 遗迹分工
- 迷雾寻路

重点不是高强度战斗，而是分工：

- 有人采集
- 有人照明
- 有人记录路线
- 有人处理突发事件
- 有人搬运
- 有人保护

这样更贴经营与探索的混合气质。

### 7.11 村社共治：终局大玩法

共治是非常激进，但也非常有桃源乡气质的终局方向。

可以做：

- 每周村社会议
- 公共建设投票
- 节庆主题选择
- 本周扶持产业
- 公共仓库分配
- 新住户邀请
- 世界事件响应
- 乡约条款制定

这会让联机不只是互相帮忙，而是共同塑造一段生活秩序。

## 8. UI 入口想象：玩家从哪里进入联机世界

如果联机做大，入口不能只是主菜单多一个“多人模式”。

它应该自然长进游戏世界里。

### 8.1 主菜单入口

主菜单可以有：

- 我的庄园
- 好友来访
- 邻里动态
- 今日节会
- 村社公告
- 热门庄园

这几个入口不应该像外部系统，而像游戏世界的日常消息。

### 8.2 游戏内入口

游戏内可以有这些固定入口：

- 邮箱：书信、礼物、邀请
- 交流大厅：公开讨论、攻略、公告
- 邻里牌：好友、邻里、求助单
- 村社祠堂：共建、投票、公共目标
- 集市：寄售、交换、节庆摊位
- 码头 / 驿站：来访、同游、远征房间
- 节庆会场：实时节庆活动

这样联机不会像一个外挂页，而像桃源乡世界的一部分。

### 8.3 顶层信息结构

我会把联机 UI 分成五个主面板：

- `邻里`
- `庄园`
- `委托`
- `节会`
- `村社`

每个面板只负责一类心智：

- 邻里：谁和我有关
- 庄园：别人怎么看我，我怎么看别人
- 委托：我能帮谁，谁能帮我
- 节会：今天大家一起玩什么
- 村社：我们共同决定什么

### 8.4 玩家第一天看见联机时的体验

第一天上线，玩家应该看到：

1. 你的庄园现在可以公开展示了。
2. 你可以选择一个展示主题。
3. 你可以添加朋友或加入邻里。
4. 今天有一个轻量共同目标。
5. 你可以发出第一张求助单。

这比直接丢给玩家一堆系统入口更好。

## 10. 激进首发包：如果第一版就想有冲击力

如果不考虑工作量，第一版联机我不会只做好友和留言。

我会直接做一个非常有感的“邻里初开”大版本。

### 10.1 首发核心体验

玩家上线后可以做这些事：

- 创建公开庄园页
- 设置庄园展示主题
- 邀请好友
- 加入一个邻里
- 访问别人庄园
- 给别人留言
- 给别人寄一份小礼
- 发布求助单
- 接别人求助单
- 参加本周邻里共同目标
- 查看庄园热度榜
- 查看本周互助榜
- 收到来访回执

这已经足够让玩家明确感觉：

> 桃源乡从一个人的田园，变成了一群人的乡里。

### 10.2 首发最重要的 5 个画面

第一版最应该打磨的不是接口，而是这 5 个画面：

- 我的公开庄园页
- 好友庄园页
- 邻里动态页
- 求助单面板
- 本周共同目标页

这些画面决定玩家会不会相信“联机真的上线了”。

### 10.3 首发应有的情绪反馈

首发要给玩家这些反馈：

- 有人来过我家
- 有人喜欢我的布置
- 有人帮我完成了一单
- 我帮别人解决了一个问题
- 我们邻里一起推进了一点进度
- 今天发生的事明天还能看到痕迹

这比单纯的实时同屏更有留存价值。

## 13. 最大胆的几个设计

下面这些是更激进，但我觉得很有想象力的方向。

### 13.1 玩家可以成为“低频 NPC”

不是让真人一直在线，而是把玩家的公开庄园、经营风格、留言、委托变成其他玩家世界里的低频来访内容。

例如：

- 行脚商人提到某位玩家的庄园
- 任务板出现某位玩家的求助
- 邮箱收到好友的节气信
- 热门庄园成为 NPC 对话里的传闻

这样就算玩家不同时在线，也会彼此进入对方的世界。

### 13.2 庄园风格会影响别人收到的祝福

比如一个玩家长期经营花田，他给别人送祝福时可能偏向：

- 花香祝福
- 心情加成
- 关系加成

一个玩家擅长钓鱼，他的祝福可能偏向：

- 水运
- 鱼汛
- 稀有鱼概率

玩家的经营人格会变成社交能力。

### 13.3 村社可以形成产业

一个村社可以偏向：

- 渔业村社
- 药草村社
- 育种村社
- 工坊村社
- 探索村社
- 节庆村社

不同村社可以有不同公共技能、任务偏置和节庆加成。

### 13.4 共同工程会改变公共地图

比如：

- 修好码头后开放海货活动
- 建好书院后开放藏书交换
- 修好灯街后上元节内容变多
- 建好驿站后好友互访更方便

这会让共建不是捐材料领奖，而是改变世界结构。

### 13.5 节庆可以由玩家共同塑形

节日主题可投票：

- 今年中秋偏赏月
- 今年中秋偏厨艺
- 今年中秋偏灯谜
- 今年中秋偏团圆宴

同一个节日每年都可能因为玩家选择不同而变化。

## 14. 风险：激进不代表可以无视代价

这份文档不考虑工作量，但不代表不考虑后果。

### 14.1 最大风险是经济失控

任何跨玩家资源流动都会影响单人经济。

需要：

- 限量
- 冷却
- 品类限制
- 价值上限
- 服务端流水
- 异常检测

### 14.2 第二大风险是存档覆盖

联机奖励必须避免被本地旧存档覆盖。

需要：

- revision
- 领取凭证
- 幂等写入
- 补偿日志
- 当前运行态提示

### 14.3 第三大风险是社交审核

只要能留言、命名、上传，就会有审核问题。

需要：

- 举报
- 屏蔽
- 敏感词
- 图片限制
- 管理员工具
- 审计记录

### 14.4 第四大风险是系统过多导致主线变糊

联机系统很容易越做越散。

解决办法：

- 所有联机内容都归到 `庄园 / 邻里 / 委托 / 节会 / 村社`
- 玩家每天最多看到 1 个主联机目标
- 联机奖励必须回流到单人经营

### 14.5 第五大风险是实时玩法喧宾夺主

实时活动很吸引人，但不能把桃源乡变成排队房间游戏。

解决办法：

- 实时活动只在节庆、远征、特殊事件里出现
- 日常主循环仍然是经营
- 实时奖励服务于经营，而不是替代经营

## 15. 最终收口：桃源乡联机应该卖什么

如果未来真的宣传这套联机，它不应该卖：

- 大型多人在线
- 开放世界同步
- 自由交易
- 排位竞技

它应该卖：

- 你的庄园终于有人来访
- 朋友能真正帮上你的经营
- 节日不再只是系统活动
- 邻里会一起改变桃源乡
- 每个玩家都能在世界里留下生活痕迹

一句话：

> 桃源乡的联机，不是把玩家塞进同一张地图，而是让每个人的生活互相照进对方的世界。
