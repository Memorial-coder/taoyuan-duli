# 2026-04-12 游戏漏洞全面审核

## 审核范围

- 前端：`taoyuan-main/src/stores`、`src/views/game`、`src/views/dev`、`src/router`、`src/composables`
- 服务端：`server/src/routes`、`server/src/taoyuanMailbox.js`、大厅/存档/额度兑换相关逻辑
- 安全模型：存档、时间推进、离线收益、导入导出、调试入口、经济系统结算

> 说明：本次为防御视角审计，仅描述风险成因、触发路径与修复方向，不提供攻击脚本。

---

## 总结结论

该项目当前最核心的问题不是传统 Web 注入，而是：

1. **过度信任客户端状态**：前端 Pinia 状态、导入存档、组件回传结果都可被玩家本地篡改。
2. **服务端缺少权威账本/权威存档校验**：尤其是额度兑换和服务端存档接口，实际把客户端数据当真。
3. **缺少统一事务式结算层**：多个系统采用“先发奖励，再由前端标记完成/扣钱/自动存档”的模式，存在竞态与回滚不一致。
4. **存档安全模型几乎无防护**：所谓“加密”密钥写在前端，等同于玩家可以自行构造合法存档。

综合评估：**存在多个 Critical / High 级漏洞**，其中部分会导致真实服务端 quota 资产、云存档、长期经济平衡受到影响。

---

## 一、Critical 级漏洞

### 1. 额度提现接口信任客户端铜钱，存在“真钱化/额度化”风险

**严重性**：Critical  
**影响面**：账号 quota、额度兑换、依赖 quota 的后续系统

#### 证据

- `server/src/routes/api.js:625-663`
  - `POST /taoyuan/quota/export`
  - 服务端直接读取 `req.body.money`
  - 只要参数合法，就执行：`db.addQuota(username, quotaGain)`
  - **没有读取服务端权威存档，也没有扣减服务端权威铜钱余额**
- `taoyuan-main/src/views/game/WalletView.vue`
  - 搜索结果已确认存在 `handleExport` 与 `persistExchangeResult`
  - 前端流程是：接口成功 → `playerStore.setMoney(...)` 本地扣钱 → `saveStore.autoSave()` 尝试持久化

#### 风险说明

服务端已经把 quota 发出后，扣钱动作却发生在客户端内存与后续自动存档中。玩家只要在响应成功后、自动存档前中断页面流程（刷新、关闭、崩溃、断网、切回旧档等），就可能形成：

- 服务端 quota 已增加
- 本地/服务端铜钱未真正扣除

这属于典型**非原子兑换漏洞**。

#### 修复建议

- 将“扣钱 + 加 quota”移动到服务端单事务内完成
- 服务端必须持有并校验**权威铜钱余额**
- 前端不应再承担最终扣款职责

---

### 2. 服务端存档可被客户端伪造，存档“加密”不具备安全性

**严重性**：Critical  
**影响面**：本地存档、服务端存档、邮箱/大厅/联动结算、长期经济状态

#### 证据

- `taoyuan-main/src/stores/useSaveStore.ts:43`
  - `const ENCRYPTION_KEY = 'taoyuanxiang_2024_secret'`
- `taoyuan-main/src/stores/useSaveStore.ts:65-90`
  - `encrypt/decrypt/parseSaveData` 全在前端实现
- `taoyuan-main/src/stores/useSaveStore.ts:784-796`
  - `importSave()` 仅验证“能解密且结构可识别”，随后直接写入槽位
- `taoyuan-main/src/stores/useSaveStore.ts:716-721`
  - `saveToSlot()` 直接把前端构造的数据加密后保存
- `server/src/routes/api.js:682-694`
  - `POST /taoyuan/save/:slot` 直接保存 `raw`
  - **无签名、无服务端解包校验、无字段白名单、无业务约束验证**

#### 风险说明

由于密钥在客户端可见，玩家可自行构造合法密文存档，任意修改：

- 钱、票券、物品、任务状态
- 已领取标记、解锁状态、商城状态
- NPC、博物馆、公会、瀚海、村庄建设等跨系统数据

这不仅是“单机作弊”，因为服务端也会把上传后的密文当作可信存档继续使用。

#### 修复建议

- 改为服务端维护权威存档结构
- 至少增加服务端签名/MAC 校验
- 对关键字段做白名单和业务边界校验
- 关键经济字段不要信任客户端导入结果

---

### 3. 瀚海扑克 / 恶魔轮盘结算信任前端回传结果，可直接刷钱刷券

**严重性**：Critical  
**影响面**：钱包、票券、赌场奖励、相关成就/目标/经济统计

#### 证据

- `taoyuan-main/src/stores/useHanhaiStore.ts:1113-1145`
  - `startTexas()` 仅负责扣入场费并发牌
- `taoyuan-main/src/stores/useHanhaiStore.ts:1147-1168`
  - `endTexas(finalChips, tierName)` 直接使用前端传入的 `finalChips`
  - 再调用 `settleCasinoRewards('texas', trigger, finalChips)`
- `taoyuan-main/src/stores/useHanhaiStore.ts:1171-1196`
  - `startBuckshot()` 只负责扣费与初始化
- `taoyuan-main/src/stores/useHanhaiStore.ts:1198-1228`
  - `endBuckshot(won, draw)` 直接信任前端传入胜负结果

#### 风险说明

这里没有“已开始对局”的服务端或 store 内权威会话状态，也没有对结果进行校验。
因此玩家若能通过 devtools、组件 emit 篡改、直接调用 store 方法等方式调用结算函数，就可能：

- 不按真实牌局结果结算
- 反复提交高收益结算
- 在未正确开始对局的情况下直接领取奖励

#### 修复建议

- 开局时生成并保存权威对局状态
- 结算时只接受 store 内部可验证状态，不接受外部自由参数
- 加入“已开始未结算”标识和对局 nonce

---

### 4. 村庄建设捐赠不扣物品，可空手刷进度与里程碑

**严重性**：Critical  
**影响面**：村庄建设、里程碑、相关解锁链路、全局平衡

#### 证据

- `taoyuan-main/src/stores/useVillageProjectStore.ts:798-833`
  - `donateToProject(projectId, itemId, amount)` 只检查：数量>0、项目是否开放、物品是否在白名单
  - 然后直接写入：
    - `totalAmount: donationSummary.state.totalAmount + amount`
    - `donatedItems[itemId] += amount`
  - **没有调用 `removeCombinedItem()`，也没有扣库存/仓库物品**
- 同文件 `completeProject()` 中则明确有 `removeCombinedItem(material.itemId, material.quantity)`，说明该模块本应具备扣物资逻辑

#### 风险说明

玩家可以不消耗任何实际物资，就把捐赠计划推进到目标值，并继续领取相关里程碑收益或解锁后续链条。

#### 修复建议

- 捐赠前校验真实库存
- 在同一事务中完成：校验库存 → 扣物品 → 更新 donation state
- 失败时回滚

---

### 5. 商店负数购买可把消费变成加钱

**严重性**：Critical  
**影响面**：铜钱、商店、经济统计、物资结算一致性

#### 证据

- `taoyuan-main/src/stores/useShopStore.ts:1707-1717`
  - `buySeed(seedId, quantity = 1)`
  - `const totalCost = applyDiscount(seed.price) * quantity`
  - 未校验 `quantity > 0`
- `taoyuan-main/src/stores/useShopStore.ts` 搜索结果确认存在 `buyItem(itemId, price, quantity = 1)` 同类模式
- `taoyuan-main/src/stores/usePlayerStore.ts:276-283`
  - `spendMoney(amount)` 只检查 `if (money.value < amount) return false`
  - 若 `amount` 为负数，则判断恒不成立，随后执行 `money.value -= amount`
  - 即：**负数消费 = 加钱**

#### 风险说明

若外部调用这些 store 方法并传入负数数量，`totalCost` 会变为负数，最终让 `spendMoney()` 把余额增加。

#### 修复建议

- `quantity`、`price`、`amount` 全部强制 `Number.isFinite && > 0`
- `spendMoney()` 内部增加统一防线：非正数直接拒绝

## 二、High 级漏洞

### 6. 导入伪造存档可在生产环境直接作弊

**严重性**：High  
**影响面**：生产玩家体验、云存档、进度公平性

#### 证据

- `taoyuan-main/src/stores/useSaveStore.ts:784-796`
  - `importSave()` 接受外部文件内容，只要能解密并过基础结构检查即写入
- `taoyuan-main/src/views/MainMenu.vue` / `SaveManager.vue`
  - 搜索结果已确认存在调用 `saveStore.importSave(...)`

#### 风险说明

即使生产环境不开放 `/dev/late-game`，玩家仍可通过“导入存档”把任意构造的数据带入正式游戏流程。

#### 修复建议

- 禁止生产环境导入非签名存档
- 导入时加入版本签名/完整性校验/关键字段合法性约束

---

### 7. 读取存档后会重放 NPC 关系奖励，篡改状态可重复获利

**严重性**：High  
**影响面**：物品奖励、关系解锁收益、存档一致性

#### 证据

- `taoyuan-main/src/stores/useSaveStore.ts:596-598`
  - 在所有 store 反序列化后执行：
  - `npcStore.rehydrateRelationshipPerks({ grantInventoryRewards: true, emitMessages: false })`
- `taoyuan-main/src/stores/useNpcStore.ts`
  - 搜索结果确认 `rehydrateRelationshipPerks` 支持 `grantInventoryRewards`
  - 存在基于关系 benefit 发放物品的逻辑

#### 风险说明

如果玩家手工篡改 `unlockedPerks`、关系阶段或相关状态，使系统误判某些关系奖励“尚未同步”，则在读档后可能重复发放奖励。

#### 修复建议

- 为关系奖励增加不可重放的唯一领取记录
- 读档同步时区分“状态恢复”和“首次发奖”

---

### 8. 开发态 feature/balance 覆盖会随存档进入生产环境

**严重性**：High  
**影响面**：平衡、开关、后期系统表现

#### 证据

- `taoyuan-main/src/stores/useSettingsStore.ts:92-132`
  - `setFeatureOverride`、`setLateGameBalanceOverrides` 仅 DEV 可写
- `taoyuan-main/src/stores/useSettingsStore.ts:135-156`
  - `serialize()` 会把 `lateGameFeatureOverrides`、`lateGameBalanceOverrides` 写入存档
- `taoyuan-main/src/stores/useSettingsStore.ts:159-180`
  - `deserialize()` 在所有环境都会恢复这些字段

#### 风险说明

即：开发环境生成的带覆盖项存档，可被导入生产环境后继续生效，造成：

- 功能提前启用/禁用
- 数值平衡篡改
- 周目标、经济调优、后期系统行为异常

#### 修复建议

- 生产环境读档时忽略 dev-only 覆盖字段
- 将 dev override 与正式存档彻底隔离

## 三、Medium 级问题 / 架构风险

### 9. 调试路由本身在生产大概率不暴露，但整体作弊能力仍来自存档体系

**结论**：

- `taoyuan-main/src/router/index.ts:11-19`
  - `/dev/late-game` 仅在 `import.meta.env.DEV` 下注册
- `LateGameDebugView` 与相关全局 debug API 生产大概率不可直达

**但**：由于存档可伪造，生产环境依然可以通过导档实现“近似调试面板”的实际效果，因此**真正的问题不在 dev 路由，而在存档信任模型**。

---

### 10. 多个系统缺统一校验层，易出现跨模块奖励重复发放

**表现**：

- 多个 store 各自发奖、各自改状态
- 缺少统一事务层或事件结算总线
- 依赖 `serialize()/deserialize()` 快照回滚，但这只解决局部异常，不能抵御恶意输入与跨模块重入

**涉及高风险模块**：

- Quest
- Guild
- Museum
- Hanhai
- VillageProject
- Wallet / Shop

## 五、建议优先级

### P0（立刻修）

1. 修复 `/taoyuan/quota/export`，把扣钱与加 quota 合并为服务端原子事务
2. 下线“客户端可伪造服务端存档”的模型，至少加签名/MAC
3. 修复 `endTexas/endBuckshot` 的自由结算入参问题
4. 修复 `donateToProject()` 不扣物品问题
5. 给 `spendMoney/buySeed/buyItem` 加统一正数校验

### P1（尽快修）

6. 禁止生产导入未签名存档
7. 修复关系奖励重放风险
8. 生产环境忽略 dev-only balance / feature override

### P2（结构优化）

9. 建立统一经济结算层
10. 建立关键状态白名单校验与一致性校验
11. 为跨模块奖励增加幂等标记与回放保护

## 六、最终结论

该项目目前的主要漏洞来源是**“客户端可信”假设不成立**。只要玩家可以改本地状态、导入存档、直接调用前端 store 方法，就能够影响大量核心经济与进度逻辑；而服务端又在关键节点继续信任这些结果，因此风险被进一步放大。

如果只修表层 UI 或只隐藏 debug 面板，并不能真正解决问题。必须从：

- **服务端权威状态**
- **关键交易原子性**
- **存档签名与校验**
- **统一参数合法性校验**

这四个方向一起收口，才能显著降低被玩家利用的空间。
