 # 桃源游戏系统三次玩法逻辑复审报告（2026-03-31）

 > 审查方式：主代理 + 5 个 subagent 分模块交叉审计 + 关键实现抽样复核  
 > 审查基线：已包含 2026-03-30 后续多轮修复  
 > 审查范围：农场 / 温室 / 鱼塘 / 加工 / 矿洞 / 战斗 / NPC / 日结 / 商店 / 存档 / 文档口径一致性

 ---

 ## 一、复审结论摘要

 相比上一轮，项目已经补掉了大量高优先级问题，尤其是：

 - 温室规则统一、隐藏 NPC 即时发现、怪物诱饵每层限次；
 - 主线 / 委托 / 成就 / 公会 / 博物馆 / 秘密笔记 / 多条加工链路的奖励原子化；
 - 鱼塘 `returnedFishPool`、仓库出入库、旧存档兼容、`activeSlotMode`、新开局装饰重置等问题。

 但从“长期经营是否稳定、公平、可预期”角度看，当前版本仍存在一批 **仍然成立的中高优先级缺陷**，主要集中在：

 1. **边缘链路仍存在非原子入包**，会导致吞货、重复结算或日志与实际结果不一致；
 2. **自动化代操作路径**（配偶 / 雇工 / 日结）与玩家手动路径不一致，导致收益缩水；
 3. **矿洞与存档状态机** 仍有可刷取、错误回滚、坏档半加载等问题；
 4. **README / 隐私协议 / 教程文案** 与当前实现已有明显脱节。

 本轮结论：项目已从“系统性失稳”进入“高风险边界问题待收口”阶段，下一批应优先处理 **事务边界、坏档容错、自动化代操作一致性**。

 ---

 ## 二、当前仍成立的重点问题

 ## P1-1 普通农场“育种种子播种”仍绕过基础规则

 - **级别**：高
 - **问题**：普通农田的 `doPlantGeneticSeed()` / `doBatchPlantBreeding()` 没有完全复用普通播种链路，仍存在规则缺口。
 - **表现**：
   - 未统一检查凌晨 2 点强制休息；
   - 未统一检查锄头是否可用；
   - 批量育种播种推进时间后未处理 `passedOut`；
   - 规则口径与普通播种、温室育种播种不一致。
 - **涉及文件**：
   - `src/views/game/FarmView.vue`
   - 函数：`doPlantGeneticSeed`、`doBatchPlantBreeding`
 - **复现思路**：锄头升级中、体力极低或逼近凌晨 2 点时，在普通农田种育种种子，行为与普通播种不一致。
 - **修复建议**：将普通农田育种播种彻底并入统一播种事务，复用普通播种的工具 / 体力 / 熬夜 / 时间推进 / passedOut 处理。

 ---

 ## P1-2 山丘地表矿脉仍可触发“部分入包 + 不清矿脉”重复结算

 - **级别**：高
 - **问题**：`handleMineSurfaceOre()` 使用的是 `inventoryStore.addItem()`，失败时只回体力并保留矿脉，但 `addItem()` 可能已经部分写入。
 - **结果**：
   - 背包半满时可能先拿到一部分矿石；
   - 因返回 `false`，矿脉不会被清除；
   - 下次还能继续开采同一矿脉，形成重复收益。
 - **涉及文件**：
   - `src/views/game/FarmView.vue`
   - 函数：`handleMineSurfaceOre`
 - **修复建议**：改为 `canAddItem(...) + addItemExact(...)`，只有完整入包后才清除 `surfaceOrePatch` 并推进时间。

 ---

 ## P1-3 加工取消 / 拆机 / 失败回退仍有“先清状态后入包”风险

 - **级别**：高
 - **问题**：加工系统虽然已修正成品收取，但 `startProcessing()` 的回退、`cancelProcessing()`、`removeMachine()` 返还原料/成品/材料时仍大量使用 `inventoryStore.addItem()`，且不校验完整入包。
 - **风险**：
   - 背包满时原料、额外材料、已完成产物、拆机返还材料可能丢失；
   - 槽位/机器状态又会继续被清空，形成不可逆损失。
 - **涉及文件**：
   - `src/stores/useProcessingStore.ts`
   - 函数：`startProcessing`、`cancelProcessing`、`removeMachine`
 - **修复建议**：拆机 / 取消加工 / 失败回退也要改成事务式返还，必要时拒绝执行而不是直接吞料。

 ---

 ## P1-4 配偶 / 雇工代收仍走错误收获路径，巨型作物与育种收益会缩水

 - **级别**：高
 - **问题**：配偶晨间收获与雇工 `harvest` 仍直接调用 `farmStore.harvestPlot()`，没有走玩家手动收获的完整逻辑。
 - **影响**：
   - 巨型作物会被按普通单格收获处理；
   - 育种作物的额外收益、登记、经验、任务钩子可能不完整；
   - 自动化路径与玩家手动路径收益口径不一致。
 - **涉及文件**：
   - `src/composables/useEndDay.ts`
   - `src/stores/useNpcStore.ts`
   - `src/stores/useFarmStore.ts`
 - **相关函数**：配偶晨间 `harvest` 分支、`processDailyHelpers('harvest')`、`harvestPlot`
 - **修复建议**：把“代收”改为复用统一收获事务；巨型作物、育种收益、任务/成就/日志均应与手动路径一致。

 ---

 ## P1-5 仓库箱子拆卸时，满包仍可能直接吞掉箱内物品与返还材料

 - **级别**：高
 - **问题**：`confirmDismantle()` 先逐条 `addItem()` 返还箱内物品，再清空并删除箱子，再返还材料；全过程没有完整容量预检。
 - **风险**：
   - 满包时箱内物资会被部分返还、部分丢失；
   - 箱体和返还材料照样结算；
   - 属于典型“先清源后入包”吞货问题。
 - **涉及文件**：
   - `src/views/game/HomeView.vue`
   - 函数：`confirmDismantle`
 - **修复建议**：先模拟“箱内全部物品 + 返还材料”是否能完整入包；不能完整返还时禁止拆卸。

 ---

 ## P1-6 主矿洞 BOSS 戒指 / 帽子 / 鞋子仍可重复刷取

 - **级别**：高
 - **问题**：BOSS 装备补发条件仍基于“当前是否持有”而不是“历史是否已领取”。
 - **结果**：
   - 玩家首杀后卖掉/战败回滚掉对应装备；
   - 再次击杀同层 BOSS 时会再次补发；
   - 可反复刷钱或刷装备。
 - **涉及文件**：
   - `src/stores/useMiningStore.ts`
   - 函数：`handleMonsterDefeat`
 - **修复建议**：为 BOSS 装备增加独立的领取记录，而不是使用 `hasRing/hasHat/hasShoe` 作为幂等依据。

 ---

 ## P1-7 矿洞战利品回滚与临时背包不一致，仍可保留或误删物品

 - **级别**：高
 - **问题**：
   - 矿洞获得物品时 `addItem()` 可能把东西写入 `tempItems`；
   - 但战败回滚调用的是 `inventoryStore.removeItem()`，只会从主背包 `items` 删除；
   - 且 `recordItemLoot()` 是按“理论掉落”记账，不关心是否完整写入。
 - **风险**：
   - 进入临时背包的战利品可能在战败后保留下来；
   - 若本次掉落未完整入包，回滚时还可能误删玩家原先已有的同类主背包物品。
 - **涉及文件**：
   - `src/stores/useMiningStore.ts`
   - `src/stores/useInventoryStore.ts`
 - **相关函数**：`recordItemLoot`、`rollbackLootEntry`、`handleDefeat`、`addItem`、`removeItem`
 - **修复建议**：矿洞战利品要么只在完整入包后才记账，要么回滚逻辑必须同时覆盖主背包与临时背包，并记录真实写入结果。

 ---

 ## P1-8 坏档 / 脏档加载仍可能先清空当前会话，再在半途失败

 - **级别**：高
 - **问题**：`loadFromSlot()` / `importSave()` 只校验外层 envelope；`applySaveData()` 会先 `$reset()` 全部 store，再逐个 `deserialize()`。如果某个 store 在深层结构上抛错，当前游戏会先被清空，再以失败结束。
 - **涉及文件**：
   - `src/stores/useSaveStore.ts`
   - 函数：`loadFromSlot`、`importSave`、`applySaveData`
 - **修复建议**：增加“预反序列化校验”或“影子实例加载”，确认全部 store 可安全读取后再正式覆盖当前会话。

 ---

 ## P2-1 非法双戒指状态仍可从旧档 / 脏档残留，并双倍生效

 - **级别**：中高
 - **问题**：运行时 `equipRing()` 已禁止双槽同戒指 / 同 `defId`，但 `deserialize()` 不会修复旧档中的非法状态；`getEquipmentBonus()` 会对两个槽位都累计。
 - **结果**：旧档或脏档仍可保留“双戒指双倍加成”状态。
 - **涉及文件**：
   - `src/stores/useInventoryStore.ts`
   - 函数：`deserialize`、`getEquipmentBonus`
 - **修复建议**：读档时主动去重并清洗非法双戒指状态。

 ---

 ## P2-2 晨间赠物 / 知己赠礼 / 配偶做饭等链路仍有“日志成功但实际丢物”问题

 - **级别**：中
 - **问题**：`useEndDay.ts` 中仍有多处晨间赠物、知己送礼、配偶做饭、洞穴/林木被动产物直接使用 `addItem()`，未统一容量预检。
 - **影响**：背包满时日志仍可能显示“送来了/做了一份/发现了”，但物品实际部分或全部丢失。
 - **涉及文件**：
   - `src/composables/useEndDay.ts`
 - **修复建议**：把日结发物也纳入统一原子发放策略，至少做到“日志与结果一致”。

 ---

 ## P2-3 钓鱼 / 蟹笼链路仍有局部非原子入包与吞物风险

 - **级别**：中
 - **问题**：
   - `completeFishing()` 钓获成功后仍使用 `addItem()`，满包时直接丢鱼；
   - `rollTreasureChest()` 宝箱奖品直接 `addItem()`；
   - `removeCrabPot()` 先移除蟹笼实体再回背包，不检查回收是否成功；
   - `collectCrabPots()` 每日收蟹笼产物时也直接 `addItem()`。
 - **涉及文件**：
   - `src/stores/useFishingStore.ts`
   - 函数：`completeFishing`、`rollTreasureChest`、`removeCrabPot`、`collectCrabPots`
 - **修复建议**：钓获、宝箱、蟹笼回收和蟹笼日结产物全部补齐容量预检与精确入包。

 ---

 ## P2-4 美观度加成的“好感上限 +250”只在部分路径生效，关系系统上限不一致

 - **级别**：中
 - **问题**：`getFriendshipCap(state, beautyCapBonus)` 的额外上限只在 `dailyReset()` 里显式传入，大量日常互动、送礼、事件推进仍按旧上限 2500/4000 截断。
 - **结果**：同一套关系系统在“每日结算”和“平时互动”两条路径上的上限口径不一致。
 - **涉及文件**：
   - `src/stores/useNpcStore.ts`
   - 函数：`dailyReset`、`talkTo`、`giveGift`、`adjustFriendship`、`startDating`、`propose` 等
 - **修复建议**：统一从装饰系统读取当前上限加成，而不是仅在 `dailyReset()` 特判。

 ---

 ## P2-5 文档、隐私协议与当前实现存在明显冲突

 - **级别**：中高（产品一致性）
 - **主要问题**：
   1. 主菜单隐私协议写“存档不会上传至服务器”，但 UI 与实现都已支持“服务端持久化”；
   2. README 仍写背包 `20-36 格 / 单格 99`，实际为 `24 → 120 / 单格 999 / 临时背包 10`；
   3. README 仍写体力上限至 `180`，实际成长线更高；
   4. README 仍写工具“三档（基础→铁制→精钢）”，实际为四档；
   5. README 的动物总数与名单已与实现脱节。
 - **涉及文件**：
   - `README.md`
   - `src/views/MainMenu.vue`
   - `public/guide.html`
   - `public/guide-book.html`
 - **修复建议**：统一“本地 / 服务端存档”口径，全面更新 README 与 guide 中的核心规则参数。

 ## 三、优先级建议

 ### 第一优先级（建议立即处理）

 1. 普通农田育种播种规则统一；
 2. 地表矿脉事务化；
 3. 加工取消/拆机返还事务化；
 4. 配偶/雇工代收改为复用统一收获事务；
 5. 仓库拆箱前做整组容量校验；
 6. 矿洞 BOSS 装备加独立领取幂等；
 7. 矿洞战利品真实写入记账 + 临时背包回滚；
 8. 存档加载改为“先校验后覆盖”。

 ### 第二优先级（建议下一批处理）

 1. 清理非法双戒指旧档状态；
 2. 统一日结晨间发物原子化；
 3. 修复钓鱼 / 蟹笼链路的非原子入包；
 4. 统一美观度关系上限逻辑；
 5. 更新 README / guide / 隐私协议。

 ---

 ## 四、总体评价

 当前桃源项目已经从“核心玩法链路存在大面积事务缺口”进化到“少数高风险边缘路径仍未完全收口”的阶段。也就是说，主体系统已经能稳定运行，但凡是 **自动化代操作、异常容量、旧档脏档、矿洞失败回滚** 这类边界条件，仍然是最容易出问题的区域。

 如果下一轮继续收尾，我建议优先围绕两条主线推进：

 1. **把所有“来源状态变化 + 奖励写入”彻底事务化**；
 2. **把读档、战败、自动化代操作这些边界状态机做成真正可回滚、可校验、可拒绝的安全流程**。

 这两块收完后，项目的玩法可信度和维护稳定性会再上一个台阶。
