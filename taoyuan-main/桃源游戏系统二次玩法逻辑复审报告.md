# 桃源游戏系统二次玩法逻辑复审报告

> 审查时间：2026-03-30  
> 审查方式：主代理 + 5 个 subagent 分模块交叉复核 + 关键文件抽样验证  
> 审查范围：跨天结算、移动与营业时间、农场/温室/鱼塘、矿洞/战斗/NPC、存档状态机、对外规则一致性

---

## 一、复审结论摘要

本轮复审结果显示：项目相较上一版已经修掉了一批明显缺陷，例如：

- 熬夜睡觉惩罚的入睡时间读取顺序已修正；
- 跨区域移动现在会检查体力扣除是否成功；
- 隐藏 NPC 的 `forage / village` 路由 key 已与实际路由对齐；
- 矿洞感染层奖励、主矿洞 BOSS 固定奖励已加入幂等标记；
- “下一层”现在只会在成功前进时消耗时间；
- 孕期 `caredToday` 现在会拦住全部当日照料行为，而非只限制单一分支。

但从“长期经营是否稳定、公平、可预期”来看，当前版本仍存在一批**高优先级玩法逻辑缺陷**，主要集中在以下几类：

1. **资源先清状态、后入背包**，导致背包满时会直接吞货；
2. **鱼塘品种与待领取产物状态机不稳定**，既能刷图鉴，也会覆盖未领取收益；
3. **矿洞安全点与 BOSS 层重叠**，可直接绕过关键门槛；
4. **矿洞战斗时间成本失真**，可无限拖回合和反复接敌；
5. **导航与存档状态机仍有边界漏洞**，会造成营业时间绕过和自动存档写错目标；
6. **部分关系/事件系统仍存在状态前后矛盾**；
7. **文档/隐私/规则说明与实际实现仍有明显冲突**。

## 二、本轮重点确认：上一版中已不再成立的问题

以下问题经本轮复核，**当前代码中已修复或已不再复现**：

1. **熬夜睡觉体力恢复惩罚失效**  
   - `src/composables/useEndDay.ts` 已在 `nextDay()` 之前先记录 `const bedHour = gameStore.hour`。

2. **跨区域移动不检查体力扣除结果**  
   - `src/stores/useGameStore.ts` 的 `travelTo()` 现在对 `consumeStamina()` 失败直接返回。

3. **隐藏 NPC 路由名与配置不一致**  
   - `src/data/hiddenNpcs.ts` 现为 `forage / village`，与 `src/router/index.ts` 对齐。

4. **感染层奖励 / 主矿洞 BOSS 固定奖励可重复领取**  
   - `src/stores/useMiningStore.ts` 已增加 `claimedInfestedRewardFloors` 与 `claimedBossRewardFloors`。

5. **“下一层”失败也照常耗时**  
   - `src/views/game/MiningView.vue` 中仅在 `result.success` 时才 `advanceTime(ACTION_TIME_COSTS.nextFloor)`。

6. **孕期补品可同日无限刷**  
   - `src/stores/useNpcStore.ts` 的 `performPregnancyCare()` 现在在入口先统一检查 `pregnancy.value.caredToday`。

## 三、当前仍成立的核心问题清单

## 1. 温室一键收获先清地块，再按体力发奖，仍会吞作物

- **级别**：严重
- **影响**：玩家体力不足时，温室成熟作物会先被清空，后续因体力不够无法发奖，造成明确资产损失。
- **证据**：
  - `src/views/game/FarmView.vue`：`doGhBatchHarvest()` 先调用 `farmStore.greenhouseBatchHarvest(availableStamina)`；
  - 同函数后续才循环 `playerStore.consumeStamina(1)` 并 `inventoryStore.addItem(cropId, 1, quality)`；
  - `src/stores/useFarmStore.ts`：`greenhouseBatchHarvest()` 内部是逐格调用 `greenhouseHarvestPlot()`，即先改地块状态，再返回结果列表。
- **复现思路**：
  1. 温室准备多块成熟地；
  2. 只保留 1 点或少量体力；
  3. 点击温室“一键收获”；
  4. 后半段作物会被清空但无法真正入包。
- **修复建议**：
  - 改成“逐格校验体力 → 扣体力成功 → 收获地块 → 入包”；
  - 或先计算可收数量上限，仅对可支付体力的地块执行收获；
  - 若 `addItem()` 失败，也必须回滚地块状态。

---

## 2. 多条收获/领取链路都是“先清状态，后入背包”，背包满会直接吞货

- **级别**：严重
- **影响**：不只是温室，普通农地、巨型作物、采脂器、鱼塘等多个系统都存在“来源已清空，但入包失败不回滚”的问题，会在长期经营中反复造成隐性资产损失。
- **证据**：
  - `src/composables/useFarmActions.ts`：`handlePlotClick()` 在普通收获分支里先 `farmStore.harvestPlot(plotId)`，后 `inventoryStore.addItem(cropId, harvestQty, quality)`；
  - 同文件 `handleBatchHarvest()` 同样先 `harvestPlot / harvestGiantCrop`，后入背包；
  - `src/views/game/FarmView.vue`：`doHarvest()` 里巨型作物先 `farmStore.harvestGiantCrop()`，后 `inventoryStore.addItem()`；
  - 同文件 `handleCollectTapProduct()` 先 `farmStore.collectTapProduct(treeId)`，后 `inventoryStore.addItem(productId)`；
  - 同文件 `doGhHarvest()` 先 `farmStore.greenhouseHarvestPlot()`，后 `inventoryStore.addItem()`；
  - `src/views/game/FishPondView.vue`：`handleCollect()` 先 `fishPondStore.collectProducts()` 清空待领列表，再逐个 `inventoryStore.addItem()`。
- **复现思路**：
  1. 塞满主背包与临时背包；
  2. 去收普通作物 / 巨型作物 / 采脂器 / 鱼塘产物；
  3. 来源状态消失，但物品并不会完整保留。
- **修复建议**：
  - 所有收获型接口统一改为事务式顺序：**检查容量 → 发奖励成功 → 再清来源状态**；
  - 若支持“部分领取”，必须保留未成功入包的剩余内容；
  - 最好把“状态清除 + 奖励发放”封装成单一 store 事务，而不是拆到 view 层分步执行。

## 3. 鱼塘未领取产物会在次日被新结果直接覆盖

- **级别**：高
- **影响**：玩家如果某天忘记去鱼塘点击“收获”，上一天的待领取产物会在下一次日结时被新产物覆盖，属于隐性收益丢失。
- **证据**：
  - `src/stores/useFishPondStore.ts`：`dailyUpdate()` 最后直接执行 `pendingProducts.value = [...result.products]`；
  - 同文件 `collectProducts()` 是从 `pendingProducts` 一次性取出；没有任何“累加未领取产物”的逻辑。
- **复现思路**：
  1. 让鱼塘连续两天都能产出；
  2. 第一天产出后不去领取；
  3. 第二天结算后查看待领取列表，只剩第二天结果。
- **修复建议**：
  - 改为 `pendingProducts.value.push(...result.products)`；
  - 或明确设计为“过期作废”，但必须在 UI 和说明文案中强提示，否则对玩家来说就是吞收益。

## 4. 鱼塘可通过“放入 → 取出 → 再放入”无限刷新 Gen1 品种并刷图鉴

- **级别**：高
- **影响**：玩家可零成本刷鱼塘图鉴、快速凑配方亲本，破坏鱼塘的品种发现与繁殖推进节奏。
- **证据**：
  - `src/stores/useFishPondStore.ts`：`addFish()` 每次放鱼入塘都会重新随机 `getGen1BreedsForFish(fishId)`，并立刻 `discoveredBreeds.value.add(breed.breedId)`；
  - 同文件 `removeFish()` 取出时只返还基础 `fish.fishId`，不会保留 `breedId / genetics / name`。
- **复现思路**：
  1. 准备 1 条可养殖鱼；
  2. 放入鱼塘，记录品种；
  3. 立即取出再放入；
  4. 反复操作即可不断刷到新 Gen1 品种。
- **修复建议**：
  - 鱼的品种/基因应在“获得鱼物品”时就固定；
  - 至少在取出时要保留 `breedId` 与遗传数据，重新放回时恢复原鱼；
  - 图鉴不应在“放入瞬间”解锁，而应在真正获得该品种时解锁。

## 5. 洒水器不占种植位，仍可与作物叠放

- **级别**：高
- **影响**：玩家相当于白嫖种植面积，长期显著提高亩产与自动化收益，破坏农场平衡。
- **证据**：
  - `src/stores/useFarmStore.ts`：`placeSprinkler()` 仅检查该格是否已有洒水器，不要求地块不可种植；
  - 同文件 `plantCrop()` 与 `plantGeneticSeed()` 的逻辑未把“有洒水器的地块”作为硬限制；
  - `getAllWateredBySprinklers()` 的注释与实现都表明“含洒水器自身所在地块”。
- **复现思路**：
  1. 在可耕地块放洒水器；
  2. 再继续在同格种植；
  3. 该格同时承担“设备位 + 作物位”。
- **修复建议**：
  - 洒水器必须占地；
  - 或将其改成独立设施格，而不是与作物格共享 ID；
  - 对已有存档需做一次冲突数据迁移。

## 6. 商店营业时间按“出发时刻”判定，而不是“到达时刻”判定

- **级别**：中
- **影响**：玩家可以在出发时还开门、但路上耗时后已经打烊的情况下，仍进入商店/工坊页面，绕过营业时间限制。
- **证据**：
  - `src/composables/useNavigation.ts`：`navigateToPanel()` 先调用 `isShopOpen(panelKey, gameStore.day, gameStore.hour)`；
  - 同函数后续才执行 `gameStore.travelTo(panelKey)`；
  - `src/stores/useGameStore.ts`：`travelTo()` 内部会调用 `advanceTime(cost)`，即真正到达时刻已变化。
- **复现思路**：
  1. 在接近打烊前几分钟从远处出发去商圈/工坊；
  2. 出发瞬间判定为“营业中”；
  3. 实际到达时已过营业时段，但仍能进入。
- **修复建议**：
  - 营业判断改为基于“到达后时刻”；
  - 或进入后再次校验，若赶到时已打烊，则阻止进入并给出提示。

## 7. 存档模式切换与活跃槽位未绑定，自动存档可能写到错误介质

- **级别**：高
- **影响**：当前会话在本地模式下玩到一半，切到服务端模式后，跨天自动存档会直接写入“同编号的服务端槽位”；反向切换也成立。删除同编号槽位时，还可能把当前活跃槽位错误清空，导致后续自动存档静默失效。
- **证据**：
  - `src/stores/useSaveStore.ts`：`activeSlot` 只是单独的数字，没有和 `storageMode` 绑定；
  - `setStorageMode()` 只切换 `storageMode`，不迁移/重置 `activeSlot`；
  - `autoSave()` 直接对当前 `activeSlot` 调 `saveToSlot()`；
  - `deleteSlot()` 只要 `activeSlot.value === slot` 就会清空活跃槽，不区分当前存储模式；
  - `src/views/MainMenu.vue`：主菜单允许直接在“本地存储 / 服务端持久化”之间切换。
- **复现思路**：
  1. 读取本地存档 1 进入游戏；
  2. 回菜单切到服务端模式；
  3. 不重新加载新槽，直接睡觉触发自动存档；
  4. 当前进度会写入服务端 1 号槽的语义位置。
- **修复建议**：
  - 将 `activeSlot` 改为 `{ mode, slot }` 结构；
  - 切换模式时默认清空活跃槽，要求玩家重新选择/加载；
  - UI 上明确显示“当前会话来源：本地/服务端 + 槽位号”。

## 8. 新开局仍未重置装饰系统，可能继承上局的美观度与关系增益

- **级别**：中
- **影响**：新游戏可能继承上一局的装饰摆放、每日好感加成、商店折扣等状态，造成明显跨档污染。
- **证据**：
  - `src/views/MainMenu.vue`：`handleNewGame()` 会先调用 `resetAllStoresForNewGame()`；
  - `src/composables/useResetGame.ts`：该重置链中依然没有 `useDecorationStore().$reset()`；
  - `src/stores/useNpcStore.ts`：`dailyReset()` 中会读取 `useDecorationStore()` 的每日好感加成与美观度上限加成。
- **复现思路**：
  1. 在旧局中堆高装饰美观度；
  2. 不刷新页面，直接新开一局；
  3. 检查新局相关加成是否异常残留。
- **修复建议**：
  - 在 `resetAllStoresForNewGame()` 中加入 `useDecorationStore().$reset()`；
  - 新开局后做一次关键派生值校准。

## 9. 隐藏 NPC 的地点/时段发现条件仍然只在日结时检查

- **级别**：高
- **影响**：很多“你在某时段到过某地点就该触发”的邂逅条件，被错误实现成“睡觉结算那一刻你必须还在该地点且时段满足”，实际体验非常反直觉。
- **证据**：
  - `src/composables/useEndDay.ts`：`handleEndDay()` 中调用 `hiddenNpcStore.checkDiscoveryConditions()`；
  - `src/stores/useHiddenNpcStore.ts`：`evaluateCondition()` 直接读取 `router.currentRoute.value.name` 与 `gameStore.hour`；
  - `src/data/hiddenNpcs.ts`：大量条件带有 `timeRange` 与 `location`。
- **复现思路**：
  1. 白天真实到过指定地点；
  2. 当时不会立即发现；
  3. 只有睡觉时若仍满足对应页面/时段语义，才可能触发。
- **修复建议**：
  - 在“时间推进、面板切换、到达地点、关键交互完成”后都做条件检测；
  - 将“当天曾满足过条件”落成持久标志，再由日结统一确认剧情推进。

## 10. 主矿洞 BOSS 层与安全点重叠，允许直接跳过 BOSS

- **级别**：严重
- **影响**：玩家到达 20/40/60/80/100/120 层后，即使未击败 BOSS，也能把该层当成安全点保存；下次重进会从下一层开始，直接绕过 BOSS 门槛。
- **证据**：
  - `src/data/mine.ts`：主矿洞层生成逻辑里 `posInZone === 20` 会设为 `specialType = 'boss'`；同一套数据中又有 `isSafePoint: i % 5 === 0`；
  - `src/stores/useMiningStore.ts`：`goNextFloor()` 与 `leaveMine()` 都会在到达/离开安全点时更新 `safePointFloor`；再次进入矿洞时从 `baseFloor + 1` 开始。
- **复现思路**：
  1. 打到 19 层并进入 20 层；
  2. 不打 BOSS，直接离开矿洞；
  3. 再进入时从 21 层开始。
- **修复建议**：
  - BOSS 层不应被视为可保存安全点；
  - 或只有在该层 BOSS 被击败后，才允许写入安全点进度。

## 11. 矿洞战斗回合本身不消耗时间，重新接敌也不耗时

- **级别**：高
- **影响**：玩家可以在一个夜晚中无限拖战斗回合、反复试错、反复重新接敌，严重破坏矿洞的时间压力与风险设计。
- **证据**：
  - `src/views/game/MiningView.vue`：`handleCombat()` 只调用 `miningStore.combatAction(action)`，没有任何 `advanceTime()`；
  - 同文件中，“已揭示怪物/BOSS格重新交战”分支调用 `engageRevealedMonster(tile.index)` 后直接进入战斗，也没有推进时间；
  - 只有首次翻开怪物格并进入战斗时，才会 `advanceTime(ACTION_TIME_COSTS.combat)`。
- **复现思路**：
  1. 首次翻怪进入战斗后，连续攻击/防御多回合；
  2. 时钟不会变化；
  3. 逃跑后再次点已揭示怪物，也不会再消耗时间。
- **修复建议**：
  - 每个战斗回合都应推进时间；
  - 至少攻击、防御、逃跑、重新接敌都应消耗标准战斗时长；
  - 若担心节奏过慢，可降低单回合耗时，而不是完全不计时。

## 12. 已婚后仍可能触发婚前 8 心告白类心事件

- **级别**：中
- **影响**：玩家已经与 NPC 结婚后，仍可能看到婚前告白剧情，造成明显的人物关系状态矛盾。
- **证据**：
  - `src/stores/useNpcStore.ts`：`checkHeartEvent()` 只排除了“知己不触发 `_heart_8`”的情况；
  - 同函数并没有阻止 `state.married` 的 NPC 继续触发未完成的恋爱心事件。
- **复现思路**：
  1. 在某 NPC 未触发 8 心事件前直接推进婚恋主线并结婚；
  2. 之后继续对话，仍可能命中该 8 心告白事件。
- **修复建议**：
  - 对所有婚前恋爱事件加入 `!state.married` 与必要的 `state.dating` 检查；
  - 最好为心事件增加显式前置状态，而不是依赖事件 ID 命名约定。

## 四、文档与规则一致性问题（附录）

这些问题不一定立刻卡死流程，但会明显误导玩家理解规则。

### 1. 隐私协议与实际“服务端持久化”能力冲突

- **级别**：高
- **证据**：
  - `src/views/MainMenu.vue` 明确提供“本地存储 / 服务端持久化”切换；
  - 同文件隐私协议却写“存档数据不会上传至服务器”；
  - `src/stores/useSaveStore.ts` 存在完整 `server` 模式读写逻辑。
- **影响**：玩家无法准确判断存档是否上云、是否与账号绑定、隐私边界在哪里。

### 2. README 的背包规则与实现严重不一致

- **级别**：高
- **证据**：
  - `README.md`：背包写为“20-36 格，单格上限 99”；
  - `src/stores/useInventoryStore.ts`：实际是 `INITIAL_CAPACITY = 24`、`MAX_CAPACITY = 120`、`MAX_STACK = 999`、`TEMP_CAPACITY = 10`。

### 3. README 的体力成长上限与实现不一致

- **级别**：高
- **证据**：
  - `README.md`：写“体力上限提升至 180”；
  - `src/stores/usePlayerStore.ts`：`STAMINA_CAPS = [120, 160, 200, 250, 300]`，且还有 `addBonusMaxStamina()` 额外叠加。

### 4. README 的工具升级层级与实现不一致

- **级别**：中
- **证据**：
  - `README.md`：写“三档升级（基础→铁制→精钢）”；
  - `src/stores/useInventoryStore.ts`：工具 tier 实际是 `basic / iron / steel / iridium` 四档。

### 5. README 的动物数量与清单和实现不一致

- **级别**：中
- **证据**：
  - `README.md`：写“共 19 种动物”，并给出包含驼鹿等列表；

## 六、总体评价

本项目的系统广度依然非常强，农耕、鱼塘、矿洞、婚恋、隐藏内容、存档模式等模块都已具备较完整雏形。问题不在于“功能少”，而在于**系统联动一多，状态机与事务边界没有完全收紧**：一旦出现“先改状态、后发奖励”“UI 判断与真实到达状态脱节”“进度标记与模式来源未绑定”的情况，就会迅速演变成吞货、跳关、刷收益或存档写错目标。

如果以“能不能玩”来衡量，当前版本当然是可玩的；但如果以“长期经营是否稳定、公平、可预期”来衡量，建议优先消化本报告中的 **P0 / P1 项**。这批问题修完之后，整个桃源系统的可信度和可维护性都会明显上升。
