# 桃源游戏系统玩法与逻辑复审报告（2026-03-30）

> 复审方式：主代理 + 多个 subagent 分模块交叉审计  
> 审查范围：农场 / 温室 / 鱼塘 / 加工 / 矿洞 / 战斗 / 任务 / 商店 / 存档 / 家庭 / 全局状态一致性

---

## 一、复审结论

本轮复审确认：

1. 旧一轮报告中的部分 **P0/P1 问题已经修复**；
2. 但当前版本仍存在一批新的 **高优先级玩法与逻辑缺陷**，主要集中在：
   - **背包写入非原子**导致的经济套利或物品丢失；
   - **温室 / 育种动作链规则不一致**导致的低成本操作；
   - **任务提交与收集进度语义不一致**导致的“白嫖提交”；
   - **跨 store 状态管理不统一**导致的新开局污染、存档上下文错乱等问题；
   - **矿洞战斗与刷怪机制**仍有明显可利用空间。

---

## 二、当前确认的重点问题

### P1-1 背包写入非原子，导致“部分入包 + 全额退款 / 整份折现”

- 影响模块：商店购买、目标奖励、部分收获链路
- 风险：可刷钱、可刷物、或在满包时吞奖励
- 根因：`addItem()` 允许部分写入，但上层逻辑把它当作原子成功/失败接口使用

### P1-2 温室种植 / 育种 / 收获链规则不一致

- 影响模块：`FarmView.vue`
- 风险：温室行动缺少与普通农田一致的体力、耗时、熬夜或工具校验，破坏经营节奏

### P1-3 鱼塘 / 农场 / 巨型作物 / 加工成品存在“先清源后入包”风险

- 风险：当背包空间不足时，产物可能在源数据被清除后无法完整回收
- 典型场景：鱼塘收获、温室收获、普通农场收获、巨型作物收获

### P1-4 非 delivery 任务允许“进度完成但不实际交货”

- 影响模块：`useQuestStore.ts`
- 风险：玩家可先累计进度、再卖掉物品，最后仍能领奖

### P1-5 新开局 / 存档模式切换的状态隔离不足

- 影响模块：`useResetGame.ts`、`useSaveStore.ts`
- 风险：装饰系统跨档污染、切换 local/server 后仍沿用旧 activeSlot 上下文

### P1-6 矿洞仍存在可利用点

- 典型问题：
  - 战斗回合不严格绑定时间推进；
  - 怪物诱饵可在同层重复叠加刷怪；
  - 奖励 / 掉落 / 战斗节奏仍有进一步收敛空间。

---

## 三、本轮修复优先级

### 第一批（本次优先落地）

1. 为背包新增“完整放入检查 / 精确写入”能力；
2. 修复商店、目标奖励、鱼塘、温室、农场收获的非原子写入问题；
3. 修复温室与育种动作的体力/耗时/工具/熬夜规则缺失；
4. 修复非 delivery 任务提交校验；
5. 修复新开局装饰残留与存档模式切换 activeSlot 残留。

### 第二批（建议后续继续处理）

1. 矿洞战斗每回合时间成本统一；
2. 怪物诱饵增加同层次数限制或幂等保护；
3. 全面梳理 `addItem()` 调用点，补齐所有“先验证空间、后提交状态”的事务式写法；
4. 继续排查 child proposal、route/location source of truth 等跨模块一致性问题。
