# 桃源游戏系统玩法与逻辑审查报告

> 审查时间：2026-03-30  
> 审查范围：玩法规则、跨天结算、成长循环、存档流程、社交/隐藏NPC、矿洞/战斗、文档一致性  
> 审查方式：主代理 + 多个 subagent 分模块交叉审查

---

## 一、审查结论摘要

本次审查结果显示，项目整体系统覆盖面很广，玩法模块完整度较高，但在**跨天结算、资源结算顺序、状态机一致性、重复领奖、防错校验**方面存在一批会直接影响玩家体验与数值平衡的缺陷。

其中较严重的问题主要集中在：

1. **跨天/睡觉结算顺序错误**，导致熬夜惩罚失效。  
2. **鱼塘、矿洞感染层、BOSS奖励存在重复领取或可刷收益问题**。  
3. **温室批量收获会在体力不足时吞作物**，存在直接资产损失。  
4. **隐藏 NPC 发现链触发时机和地点 key 不一致**，导致部分内容反直觉甚至不可达。  
5. **若干资源/移动/养成校验缺失**，可被玩家绕过正常成本。  
6. **README / 教程 / 隐私文案与实际实现存在明显冲突**，会误导玩家。

---

## 二、问题分级说明

- **严重**：会导致玩法不可达、可稳定刷资源、核心系统失真、明显数据损失。
- **高**：会明显破坏平衡或造成强误导，影响主流程体验。
- **中**：有可见逻辑瑕疵，影响部分玩法或造成规则绕过。
- **低**：一致性或文案层问题，不一定立即破坏流程，但会积累认知成本。

---

## 三、核心问题清单

## 1. 熬夜睡觉的体力恢复惩罚基本失效

- **级别**：高
- **影响**：24:00~26:00 睡觉本应按入睡时间递减恢复体力，但实际大概率按接近正常恢复处理，导致“晚睡成本”被明显低估，结算提示也会误导玩家。
- **证据**：
  - `src/composables/useEndDay.ts`：`handleEndDay()` 中先执行 `gameStore.nextDay()`。
  - 同文件后续才读取 `const bedHour = gameStore.hour` 并传给 `playerStore.dailyReset(recoveryMode, bedHour)`。
  - `src/stores/useGameStore.ts`：`nextDay()` 内部会把 `hour.value = DAY_START_HOUR`，即重置为 6 点。
  - `src/stores/usePlayerStore.ts`：`dailyReset()` 的 `late` 分支明确依赖 `bedHour` 计算恢复比例。
- **结论**：结算使用的是“次日早晨 6 点”，不是“昨晚实际入睡时间”。
- **复现思路**：
  1. 把时间拖到 25:00 左右；
  2. 主动睡觉而非昏倒；
  3. 观察体力恢复与日志，恢复比例明显偏高。

---

## 2. 跨区域移动未校验体力是否真的扣除成功

- **级别**：中
- **影响**：玩家在体力不足时仍能正常跨区域移动并推进时间，相当于白嫖移动成本，破坏体力-时间资源约束。
- **证据**：
  - `src/stores/useGameStore.ts`：`travelTo()` 中调用 `playerStore.consumeStamina(staminaCost)` 后，**没有检查返回值**，仍继续 `advanceTime(cost)` 并切换地点组。
  - `src/stores/usePlayerStore.ts`：`consumeStamina()` 在体力不足时会返回 `false`。
- **结论**：移动成功与否没有绑定体力扣除结果。
- **复现思路**：
  1. 把体力降到低于跨区所需值；
  2. 从农场切到矿洞/瀚海等区域；
  3. 会照常到达，且时间照常推进。

---

## 3. 新开局不会重置装饰系统，可能继承旧局美观度/折扣加成

- **级别**：中
- **影响**：在同一前端会话中，新游戏可能继承上一局装饰摆放状态，造成非预期的美观度、NPC 好感加成、商店折扣等跨档污染。
- **证据**：
  - `src/views/MainMenu.vue`：`handleNewGame()` 调用 `resetAllStoresForNewGame()`。
  - `src/composables/useResetGame.ts`：重置了大量 store，但**未调用 `useDecorationStore().$reset()`**。
  - `src/stores/useDecorationStore.ts`：`beautyScore`、`dailyFriendshipBonus`、`shopDiscountBonus` 都依赖 `owned/placed` 状态。
- **结论**：新局初始化链路遗漏装饰 store。

---

## 4. 鱼塘产物可被“日结自动入包 + 手动收取”重复领取

- **级别**：严重
- **影响**：玩家可稳定复制鱼塘日产品，直接破坏经济循环。
- **证据**：
  - `src/stores/useFishPondStore.ts`：`dailyUpdate()` 会把产物写入 `result.products`，同时执行 `pendingProducts.value = [...result.products]`。
  - `src/composables/useEndDay.ts`：鱼塘日结部分直接把 `pondResult.products` 加入背包。
  - `src/views/game/FishPondView.vue`：`handleCollect()` 又调用 `collectProducts()` 领取 `pendingProducts`。
- **结论**：同一批产物被结算两次：一次自动入包，一次手动收取。
- **复现思路**：
  1. 让鱼塘进入可稳定产出状态；
  2. 睡觉进入次日；
  3. 背包先收到产物，再到鱼塘页面点击“收获”还能再领一次。

---

## 5. 温室一键收获先清空地块，再按体力发奖，体力不足会吞作物

- **级别**：严重
- **影响**：玩家可能直接损失已成熟作物，属于明确的数据损失问题。
- **证据**：
  - `src/stores/useFarmStore.ts`：`greenhouseBatchHarvest()` 会先逐格调用 `greenhouseHarvestPlot()`，直接把成熟地块变成已收获状态。
  - `src/views/game/FarmView.vue`：`doGhBatchHarvest()` 先拿到 `results = farmStore.greenhouseBatchHarvest()`，之后才逐个执行 `playerStore.consumeStamina(1)`；若体力不足则 `break`。
- **结论**：扣体力与清地块顺序反了，导致后续未发奖作物已被清除。
- **复现思路**：
  1. 温室准备多块成熟地；
  2. 只保留 1 点体力；
  3. 点击“一键收获”；
  4. 实际只拿到前几份作物，其余成熟地块被清空。

---

## 6. 洒水器不占地，可与作物叠放，等于白送格子

- **级别**：高
- **影响**：显著提高农场有效面积和产量，长期破坏农场平衡。
- **证据**：
  - `src/stores/useFarmStore.ts`：`placeSprinkler()` 只记录 `sprinklers`，不改变地块状态。
  - 同文件 `plantCrop()` / `plantGeneticSeed()` 没有检查该格是否已放洒水器。
  - `getAllWateredBySprinklers()` 还把洒水器自身格子算入浇水范围。
- **结论**：同一地块可同时作为“洒水器位 + 种植位”。

---

## 7. 隐藏 NPC 发现仅在日结时检查，导致时间/地点条件与实际玩法脱节

- **级别**：高
- **影响**：玩家即使在白天/黄昏真实到过指定地点，也可能因为**睡觉时不在那个页面或时段**而无法推进发现链，触发逻辑非常反直觉。
- **证据**：
  - `src/composables/useEndDay.ts`：`handleEndDay()` 中才调用 `hiddenNpcStore.checkDiscoveryConditions()`。
  - `src/stores/useHiddenNpcStore.ts`：`evaluateCondition()` 直接读 `router.currentRoute.value.name`、`gameStore.hour`、`gameStore.day`。
- **结论**：很多“邂逅类”条件变成了“睡前那一刻必须满足”，不是“当天曾经满足”。

---

## 8. 隐藏 NPC 部分地点 key 与真实路由名不一致，导致内容不可达

- **级别**：严重
- **影响**：至少部分月兔、狐仙、归女相关步骤会永久无法触发。
- **证据**：
  - `src/data/hiddenNpcs.ts`：条件中使用了 `panel: 'foraging'`、`panel: 'npc'`。
  - `src/router/index.ts`：真实路由名是 `forage`、`village`。
  - `src/stores/useHiddenNpcStore.ts`：`evaluateCondition()` 对 `routeName === cond.panel` 做严格相等判断。
- **结论**：配置和实际路由名称不一致，条件永远为假。

---

## 9. 孕期“补品”可以同日无限刷，`caredToday` 实际只限制了休息

- **级别**：高
- **影响**：玩家可在一天内通过补品快速堆高照料分，显著稀释孕期风险和照护策略。
- **证据**：
  - `src/stores/useNpcStore.ts`：`performPregnancyCare(action)` 中只有 `case 'rest'` 显式判断 `pregnancy.value.caredToday`。
  - 同函数尾部统一执行 `pregnancy.value.caredToday = true`。
  - 但 `case 'supplement'` 分支未见基于 `caredToday` 的拦截。
- **结论**：字段语义与实际限制范围不一致，形成可重复利用漏洞。

---

## 10. 感染层通关奖励可被炸弹重复触发

- **级别**：严重
- **影响**：可对刷感染层的清剿奖励，直接破坏矿洞经济和掉落平衡。
- **证据**：
  - `src/stores/useMiningStore.ts`：`handleMonsterDefeat()` 中，当满足击杀条件时会发放 `getInfestedClearRewards(activeFloorNum)`。
  - 同文件 `useBombOnGrid()` 中再次存在 `getInfestedClearRewards(activeFloorNum)` 的发放逻辑。
  - 搜索结果显示两处都没有“本层感染奖励已领取”标记。
- **结论**：感染层奖励缺少幂等保护。

---

## 11. 已首杀的主矿洞 BOSS 仍可反复刷固定金钱/矿石奖励

- **级别**：高
- **影响**：玩家可通过重复挑战弱化 BOSS 稳定刷铜钱和矿石，削弱长期成长曲线。
- **证据**：
  - `src/stores/useMiningStore.ts`：`handleMonsterDefeat()` 中存在 `BOSS_MONEY_REWARDS[currentFloor.value]` 与 `BOSS_ORE_REWARDS[currentFloor.value]` 发奖逻辑。
  - 同模块审查结果显示，BOSS 首杀后会生成弱化版本，但固定奖励未被首杀条件约束。
- **结论**：BOSS 奖励发放缺乏“首次击败”保护。

---

## 12. 矿洞死亡惩罚只回滚 `sessionLoot`，装备掉落和早期高价值物收益保护过强

- **级别**：高
- **影响**：
  - 战斗中获得的武器/戒指/帽子/鞋子可能不会随着死亡回退；
  - “丢一半战利品”按 `sessionLoot.pop()` 回滚，会优先删除最后拿到的物品，玩家可通过捡低价值物垫刀保护高价值物。
- **证据**：
  - `src/stores/useMiningStore.ts`：`sessionLoot` 仅记录部分物品掉落。
  - 同文件 `handleDefeat()`：`const lostCount = Math.ceil(sessionLoot.value.length / 2)`，随后循环 `sessionLoot.value.pop()` 并 `inventoryStore.removeItem(...)`。
  - 搜索结果也表明武器/戒指/帽子/鞋子是通过收藏类接口入手，不在 `sessionLoot` 结构中体现。
- **结论**：死亡惩罚不完整，且容易被逆向利用。

---

## 13. “下一层”失败时也会照常消耗时间

- **级别**：中
- **影响**：误触或失败操作会白白耗时，严重时可能导致凌晨昏倒。
- **证据**：
  - `src/views/game/MiningView.vue`：`handleNextFloor()` 无论 `miningStore.goNextFloor()` 返回成功还是失败，都会执行 `gameStore.advanceTime(ACTION_TIME_COSTS.nextFloor)`。
- **结论**：时间消耗没有与操作成功绑定。

---

## 14. README、主菜单隐私协议、实际存档实现三者互相矛盾

- **级别**：高
- **影响**：玩家无法明确判断存档是否只保存在本地、是否会上云、是否依赖账号、是否会传到服务端，影响隐私信任与使用决策。
- **证据**：
  - `README.md`：写有“localStorage + AES 加密”“支持导入导出、WebDAV 云同步”。
  - `src/views/MainMenu.vue`：存在“本地存储 / 服务端持久化”切换按钮；隐私协议却写“存档数据不会上传至服务器”。
  - `src/stores/useSaveStore.ts`：明确存在 `storageMode === 'server'` 的服务端存档逻辑。
- **结论**：文档、协议、实现口径不统一，属于高风险产品一致性问题。

---

## 15. README 的背包规则与实际实现严重不一致

- **级别**：高
- **影响**：误导玩家对前期资源压力、扩包价值、仓储设计的判断。
- **证据**：
  - `README.md`：写“背包 20-36 格，单格上限 99”。
  - `src/stores/useInventoryStore.ts`：`INITIAL_CAPACITY = 24`，`MAX_CAPACITY = 120`，`MAX_STACK = 999`。
- **结论**：核心数值规则与对外描述不一致。

---

## 16. README 的动物名单/数量与代码定义不一致

- **级别**：中
- **影响**：收集预期、攻略、百科认知会与实际内容对不上。
- **证据**：
  - `README.md`：写“共 19 种动物”，牧场列举中有“驼鹿”等。
  - 代码审查显示 `src/data/animals.ts` 的实际定义与 README 列表不完全一致，并包含 README 未强调的动物类型。
- **结论**：宣传内容和实际实现存在偏差。

## 四、问题分布总结

### 1. 结算顺序类问题

- 熬夜惩罚失效；
- 鱼塘双重领取；
- 温室批量收获先清地后扣体力；
- 矿洞失败操作仍推进时间。

**共性原因**：很多逻辑采用“先改状态，再发奖励/扣成本”的顺序，缺少事务性保护。

### 2. 状态机一致性问题

- 隐藏 NPC 发现使用“睡前瞬时状态”代替“当天达成状态”；
- 隐藏 NPC 地点 key 与真实路由不一致；
- 孕期 `caredToday` 语义与分支限制不一致。

**共性原因**：条件系统和 UI/路由/日结算之间缺少统一状态抽象。

### 3. 可重复刷收益问题

- 鱼塘重复领奖；
- 感染层奖励重复；
- BOSS 固定奖励反复刷；
- 洒水器叠放抬高产能。

**共性原因**：缺少“已领取”“已结算”“占位唯一性”等幂等标记。

### 4. 产品一致性问题

- 存档与隐私文案冲突；
- README 背包规则不一致；
- README 动物定义不一致。

**共性原因**：设计演进后，文档未同步回收。

## 五、优先修复建议

### P0（建议最优先处理）

1. 修复 `handleEndDay()` 中睡眠时间读取顺序；
2. 修复鱼塘双重结算；
3. 修复温室一键收获吞作物；
4. 给感染层奖励、BOSS奖励加“本层已领”幂等标记；
5. 修复隐藏 NPC 错误路由名。

### P1（高优先级）

1. 移动前强校验体力扣除是否成功；
2. 洒水器必须占格或禁止与作物重叠；
3. 孕期照料操作统一做每日次数校验；
4. 矿洞死亡惩罚补齐装备/特殊战利品回滚策略；
5. “下一层”“特殊操作”统一改为成功后再扣时间。

### P2（文档与体验层）

1. 统一 README / 教程 / 隐私协议 / 实际 UI 的存档口径；
2. 回收 README 中过时的背包、动物等规则描述；
3. 为隐藏 NPC 发现链增加日志或条件提示，降低黑箱感。

---

## 六、总体评价

桃源项目的优点非常明显：系统多、玩法广、数据量大、模块划分也较清晰，已经具备一款复杂模拟经营游戏的雏形。但正因为系统间联动很多，**一旦结算顺序、奖励幂等、状态条件校验稍有偏差，就会在长流程中迅速放大成平衡漏洞或玩家资产损失**。

如果只看“能不能玩”，当前版本是可玩的；但如果从“长期经营是否稳定、公平、可信”来看，仍建议先集中修复本报告中的 **P0/P1 问题**，再继续扩玩法，否则后续内容越多，排查和回归成本会越高。
