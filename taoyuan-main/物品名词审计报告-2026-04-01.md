## 2. 权威物品定义结论

### 2.1 权威物品词典

当前项目中，**正式物品定义的权威来源**是：

- `taoyuan-main/src/data/items.ts`

该文件通过：

- `export const ITEMS: ItemDef[] = [...]`
- `export const getItemById = (id: string): ItemDef | undefined => { ... }`

统一提供物品注册表与 ID 查询能力。

### 2.2 名称显示的通用机制

项目大量界面使用如下模式显示物品名：

```ts
getItemById(itemId)?.name ?? itemId
```

这意味着：

- 如果 `itemId` 已在 `ITEMS` 中注册，则显示中文名；
- 如果未注册，则**直接把原始字符串显示到界面**。

因此，界面出现原始 ID（例如 `mill_fish_feed`）通常说明：

1. 这个字符串不是正式物品 ID；或
2. 上游把“配方 ID / 内部标识”误传给了通用物品名显示函数。

## 3. 已确认的异常项

## 3.1 `mill_fish_feed`

### 结论

`mill_fish_feed` **不是正式物品 ID**，而是**加工配方 ID**。

### 证据

#### A. 在正式物品词典中不存在

文件：`taoyuan-main/src/data/items.ts`

- 已定义：`fish_feed`
- **未定义**：`mill_fish_feed`

也就是说，`mill_fish_feed` 不能作为正式背包物品被识别。

#### B. 在加工系统中它是配方 ID，不是产物 ID

文件：`taoyuan-main/src/data/processing.ts`

存在如下配方：

- `id: 'mill_fish_feed'`
- `machineType: 'mill'`
- `outputItemId: 'fish_feed'`

这说明：

- 内部配方名：`mill_fish_feed`
- 真正产出的正式物品：`fish_feed`

#### C. 正常收取时放入背包的是 `fish_feed`

文件：`taoyuan-main/src/stores/useProcessingStore.ts`

`collectProduct()` 收取产物时，写入背包使用的是：

```ts
recipe.outputItemId
```

而不是 `recipe.id`。

对于该配方来说，实际入包的是：

- `fish_feed`

不是：

- `mill_fish_feed`

#### D. 背包系统会拦截未注册物品

文件：`taoyuan-main/src/stores/useInventoryStore.ts`

`addItem()` / `addItemExact()` 在入包前会先执行：

```ts
if (!getItemById(itemId)) return false
```

因此，若某处真的试图把 `mill_fish_feed` 作为正式物品放入背包，当前逻辑会直接失败。

### 风险判断

`mill_fish_feed` 的问题更像是：

- **内部配方 ID 泄漏到 UI 展示层**

而不是：

- 当前主链路真的把它当作正式可收取物品发给玩家。

---

## 3.2 `recycle_fish_feed`

### 结论

`recycle_fish_feed` 与 `mill_fish_feed` 性质相同，**也不是正式物品 ID**，而是**加工配方 ID**。

### 证据

#### A. 在正式物品词典中不存在

文件：`taoyuan-main/src/data/items.ts`

- 已定义：`fish_feed`
- **未定义**：`recycle_fish_feed`

#### B. 在加工系统中是回收机配方 ID

文件：`taoyuan-main/src/data/processing.ts`

存在如下配方：

- `id: 'recycle_fish_feed'`
- `machineType: 'recycler'`
- `outputItemId: 'fish_feed'`

即：

- 配方 ID：`recycle_fish_feed`
- 实际正式产物：`fish_feed`

#### C. 正常收取仍然写入 `fish_feed`

文件：`taoyuan-main/src/stores/useProcessingStore.ts`

收取逻辑同上，最终入包仍然是：

- `fish_feed`

而不是 `recycle_fish_feed`。

### 风险判断

这也是一个**内部配方 ID 与正式物品 ID 混用的显示层风险**。

## 4. 与截图问题直接相关的判断

### 4.1 为什么界面会显示 `mill_fish_feed`

结合当前代码，截图里出现 `mill_fish_feed`，最可能的原因不是“这个物品真能被正常入包”，而是：

- 某个界面把 **recipeId** 当成 **itemId** 去显示；
- 然后命中通用兜底逻辑 `getItemById(id)?.name ?? id`；
- 因为 `mill_fish_feed` 没有注册，于是原样显示出来。

### 4.2 当前代码里已经存在专门兜底

文件：`taoyuan-main/src/views/game/ProcessingView.vue`

当前页面已经专门加入了：

```ts
const getSafeRecipeOutputName = (recipeId: string): string => {
  if (recipeId === 'fish_feed' || recipeId === 'mill_fish_feed' || recipeId === 'recycle_fish_feed') {
    return getItemById('fish_feed')?.name ?? '鱼饲料'
  }
  return getRecipeOutputName(recipeId)
}
```

这说明开发过程中已经意识到：

- `mill_fish_feed` / `recycle_fish_feed` 这类内部配方 ID 可能被误展示给玩家。

因此，截图现象与代码现状**高度吻合**。

## 7. 当前确认的风险级别

## 7.1 已确认问题

### P1：内部配方 ID 泄漏到 UI

已确认存在以下内部 ID：

- `mill_fish_feed`
- `recycle_fish_feed`

它们：

- 不是正式物品；
- 未注册到 `ITEMS`；
- 一旦进入通用名称显示逻辑，就会原样显示给玩家。

### 影响

- 玩家看到原始 ID，误以为物品异常；
- 某些页面若真的把该值当正式物品处理，可能造成“按钮文案异常 / 收取项显示异常 / 旧存档状态异常”；
- 但从当前主链路静态代码看，**尚不足以证明会稳定导致购买失败或收取失败**。

## 9. 综合结论

### 结论一句话版

截图里的 `mill_fish_feed`，本质上是**内部加工配方 ID 泄漏到 UI**，不是正式物品定义；当前代码中它**不应作为正式可购买 / 可收取物品存在**。

### 更完整的结论

1. 项目中正式物品词典以 `src/data/items.ts` 为准；`fish_feed` 已定义，`mill_fish_feed` / `recycle_fish_feed` 未定义。  
2. `mill_fish_feed` / `recycle_fish_feed` 在 `src/data/processing.ts` 中属于**配方 ID**，其真实产物都是 `fish_feed`。  
3. `useProcessingStore.collectProduct()` 当前按 `outputItemId` 发放产物，因此主收取链路应当发的是 `fish_feed`。  
4. 商店、礼包、旅行商人等购买链路当前卖的也是 `fish_feed`，未发现把 `mill_fish_feed` 当正式商品售卖。  
5. 因此，本轮确认的核心问题是：**UI 名称回退 + 内部 ID 泄漏**，而不是主购买/主收取逻辑全面失效。  

---

## 10. 后续建议（仅建议，不在本次执行）

如需后续修复，建议优先做以下几类处理：

1. 全面排查所有“把 recipeId 当 itemId 显示”的界面；
2. 给加工配方与正式物品建立统一的显示映射，不再依赖 `?? id`；
3. 对旧存档 / 旧状态做一次清洗，避免残留 `mill_fish_feed` 之类的内部标识；
4. 增加一轮自动校验：凡发包 / 入包 / 商店发货的 itemId 必须能被 `getItemById()` 命中。
