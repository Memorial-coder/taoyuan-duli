# 0518 联机状态与结算原则

## 1. 核心边界

联机层统一分成四类状态，不允许混写：

- `个人存档`
- `联机事件`
- `房间状态`
- `结算凭证`

这四类对象的职责和写入规则不同，后续所有联机任务必须复用这一层边界。

## 2. 个人存档

### 2.1 定义

个人存档是玩家单人经营世界的权威数据。

包含但不限于：

- 日期、天气、地点、体力
- 农场、背包、钓鱼、育种、NPC 关系
- 单人任务推进
- 本地演出和即时 UI 状态的可持久化部分

### 2.2 写入原则

- 个人存档只通过既有服务端存档链路或本地存档链路写回。
- 联机层不能直接共享整份个人存档给别的玩家。
- 联机玩法对个人成长的影响，必须经过结算凭证再回写。

### 2.3 冲突原则

- 服务端已确认的个人存档版本高于本地旧快照。
- 本地待同步副本只在“同一账号、同一槽位、同一修订上下文”里生效。
- 如果联机奖励已经写入服务端，旧本地快照不能直接覆盖它。

## 3. 联机事件

### 3.1 定义

联机事件是跨玩家互动的过程记录，不等于奖励结果，也不等于完整房间状态。

例如：

- 好友请求
- 留言
- 求助单发布
- 送礼发起
- 交付提交
- 村社投票
- 世界进度推进

### 3.2 写入原则

- 联机事件统一服务端权威。
- 事件记录必须带 `eventId`、`actorId`、`targetId`、`domain`、`action`、`createdAt`。
- 事件可以失败、撤销或过期，但不能直接视为“奖励已经发放”。

### 3.3 审计原则

- 所有跨玩家资源流动型事件都必须留痕。
- 事件日志需要支持后续按玩家、活动、凭证、房间回查。

## 4. 房间状态

### 4.1 定义

房间状态只承担实时活动过程，不承担个人存档权威，也不承担最终奖励落账。

房间里存的是：

- 房间成员
- Ready 状态
- 当前阶段
- 实时动作
- 暂停、断线、重连信息
- 待结算摘要

### 4.2 写入原则

- 房间状态服务端权威，优先放运行时存储。
- 房间关闭后只保留必要快照和结算结果，不长期把整段实时状态塞回个人存档。
- 房间状态只在 `running/paused/settling` 等生命周期内存在。

### 4.3 断线原则

- 断线重连恢复的是房间上下文，不是重新发奖。
- 房间重连必须依赖 `roomId + memberId + sessionToken` 这类唯一上下文。

## 5. 结算凭证

### 5.1 定义

结算凭证是跨玩家奖励和联机回流的唯一落账依据。

它不是简单日志，而是可以被核验、补发、去重、审计的正式对象。

### 5.2 最小字段

- `receiptId`
- `idempotencyKey`
- `sourceDomain`
- `sourceEventId` 或 `sourceRoomId`
- `sourcePlayerId`
- `targetPlayerId`
- `targetSlot`
- `rewardPayload`
- `status`
- `createdAt`
- `expiresAt`
- `persistedAt`
- `compensationState`

### 5.3 状态建议

- `created`
- `pending_persist`
- `persisted`
- `claimed`
- `expired`
- `compensated`
- `reversed`

## 6. 跨玩家奖励幂等规则

### 6.1 幂等 key 组成

统一使用稳定来源生成 `idempotencyKey`，至少包含：

- 来源域
- 来源对象 ID
- 目标玩家 ID
- 目标槽位
- 奖励版本

### 6.2 幂等原则

- 同一 `idempotencyKey` 只能成功落账一次。
- 重试可以重复请求，但不能重复发奖。
- 任何“客户端重复点击”“网络超时重发”“断线后补提交”都只允许命中同一凭证。

### 6.3 房间与委托额外规则

- 多人房间奖励按 `roomId + memberId + settlementVersion` 去重。
- 委托交付按 `orderId + contributorId + deliveryIndex` 去重。
- 礼物领取按 `parcelId + receiverId` 去重。

## 7. 本地状态与服务端状态冲突优先级

统一优先级从高到低如下：

1. 服务端已确认的结算凭证状态
2. 服务端权威联机事件 / 房间状态
3. 当前服务端存档修订号
4. 本地待同步副本
5. 本地纯展示缓存

解释：

- 结算凭证一旦已 `persisted`，本地旧档不能把它抹掉。
- 本地 UI 可以先预演，但最终结果以服务端确认态为准。
- 展示缓存只用于提速和离线回看，不具备权威性。

## 8. 断线、重连、重试、补偿共通策略

### 8.1 断线

- 断线时优先保留待提交动作草稿和未确认凭证引用。
- 客户端只提示“待确认”，不直接提示“已到账”。

### 8.2 重连

- 重连后先拉服务端房间状态和结算凭证状态，再恢复页面。
- 若房间已进入 `settling/closed`，客户端改走结果回读而不是重放动作。

### 8.3 重试

- 重试统一复用原请求的 `idempotencyKey`。
- 服务端若已处理成功，应直接返回已存在结果，而不是再次执行。

### 8.4 补偿

- 补偿只能基于可追溯凭证或明确失败记录触发。
- 补偿也要生成新的补偿凭证，并关联原凭证 ID。
- 补偿记录必须进入 `taoyuanAuditRuntime` 可查询范围。

## 9. 对后续阶段的强约束

- `L04` 房间状态机必须建立在本文件的房间边界之上。
- `L30-L34` 委托交付与结算必须使用结算凭证，而不是直接改双方存档。
- `L40-L44` 书信与礼物要复用“事件 + 凭证 + 补偿”结构。
- `L60-L65` 节会奖励必须按房间成员逐人生成凭证，不允许客户端自报结果。
- `L110-L113` 反作弊与补偿工具必须围绕凭证和审计对象实现。
