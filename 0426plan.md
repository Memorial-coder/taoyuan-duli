# 0426 规划：行旅图体验收口与移动端适配

## 0. 文档定位

`0425plan.md` 已经把行旅图的北极星目标说得很清楚了：它不该只是“区域结算页”，而要成为一个有 **出发感、途中感、风险感、回城感** 的探索主舞台。

结合 `CHANGELOG.md` 中 2026-04-25 与 2026-04-26 的更新，可以判断当前阶段已经完成了两件关键事：

- 玩法骨架已经从“一键完成”升级成了“多阶段远征 + 旅后处理”。
- 正式入口已经开始收口，调试按钮、未开放误导入口和静默失败都在往玩家可理解的方向收。

因此，`0426plan` 不宜继续把重点放在“再补多少条路线 / 再补多少事件”，而应该转向：

1. 让玩家更快看懂当前该做什么。
2. 让回城后的收益与承接更像“办事单”，而不是“说明书”。
3. 让手机上真的能顺手玩，而不是把桌面内容折叠一下继续塞进去。

## 1. 当前阶段判断

### 1.1 已经做对的部分

- 行旅图已经不再只是“点按钮拿奖励”，而是开始有远征筹备、途中推进、扎营、撤退、旅程日志、旅后处理。
- 回城后已经能看到资源回流、推荐去向和跨系统承接，方向是对的。
- 锁区预览、入口收口、禁用原因提示已经明显减少“能看到但不能玩”的错位体验。
- `useRegionMapStore.ts` 已经把 `frontierDigest / activeSession / journeyHistory / settlementState` 这几层状态搭起来，说明后续做体验收口有抓手，不需要重写底层。

### 1.2 当前主要矛盾

当前的主要问题已经不是“没有系统”，而是“系统太多、信息太散、手机太挤”。

更具体地说：

- 玩法上已经开始复杂起来，但玩家当前看到的是很多并列信息块，而不是一条很清晰的行动路径。
- 回城承接已经存在，但仍偏文字化、说明化，距离“做完这一趟我下一步该去哪”还有半步。
- 移动端已经加了不少折叠态，但整体仍更像“桌面内容压缩版”，不是“移动端任务流重排版”。

### 1.3 0426 的阶段目标

本轮建议把阶段目标收成一句话：

> 让玩家在 30 秒内看懂“这周去哪、为什么去、现在点哪里、回来后先去哪里”，并且能在手机上单手完成一次出发与一次回城承接。

## 2. 新玩法用户视角待优化点

### 2.1 首屏仍然更像“系统总览”，不像“准备出发”

- 现状依据：`RegionMapView.vue` 首屏同时承载全局摘要、远征筹备、区域切换、区域卡、卷轴路网、事件、首领、承接、历史与资源台账；`TopGoalsPanel.vue` 又在页面最上方先给一轮周目标信息。
- 用户问题：玩家进入行旅图后，第一反应仍然是“先读很多块说明”，而不是“先做一个明确选择”。
- 本轮建议：
  - 默认只突出一个“当前建议动作”，优先级顺序为：进行中远征 > 本周焦点区域 > 最新回城办事单。
  - 把全局 digest、资源总览、历史回看下沉到二级展开区，不抢首屏。
  - 行旅图首屏只保留三件事：去哪、为什么去、怎么出发。
- 预期效果：玩家打开页面后先进入行动状态，而不是先进入阅读状态。

### 2.2 “本周该去哪、为什么去、做完去哪”仍分散在多个模块

- 现状依据：当前导向信息分散在 `TopGoalsPanel`、`frontierDigest`、区域卡、路线卡、旅后处理和承接卡里；每块都在说一点，但没有一个地方把整条链讲完。
- 用户问题：玩家知道“这里信息很多”，但不一定知道“眼下最值得做的是哪一条”。
- 本轮建议：
  - 在行旅图页内新增一个强主导的“本周建议卡”，固定包含：推荐区域、推荐路线、主要回报、建议回城去向。
  - 把 `TopGoals` 的周目标摘要和 `frontierDigest` 的区域摘要在文案上对齐，避免同一周出现两个主线口径。
  - 回城后把“下一步建议”直接改写为一张 1-3 条的办事单，而不是多块说明并列。
- 预期效果：从出发前到回城后，玩家始终只需要跟一条清晰主线走。

### 2.3 路线的收益、成本和适用场景仍然偏“字多、结论少”

- 现状依据：路线卡已经有认知、体力、耗时、熟路、传闻、同行合同、承接提示等很多字段，但这些信息仍然需要玩家自己拼出结论。
- 用户问题：玩家能看见很多状态，却不容易快速判断“这条更适合冲资源、冲情报，还是冲首领前置”。
- 本轮建议：
  - 给每条路线补一个更扫读型的结论区：`适合干什么 / 主要带回什么 / 风险高低 / 是否适合自动巡行`。
  - 把“承接提示”从长句改成结构化摘要，例如“主要去向：商圈 / 任务板”“推荐时机：回城立即处理”。
  - 对新手阶段优先展示结论，对熟悉阶段再逐步放出详细信号。
- 预期效果：玩家选路线时是“比较方案”，不是“阅读说明再脑补方案”。

### 2.4 术语密度偏高，学习成本开始抬头

- 现状依据：当前同时出现 `情报 / 勘明 / 熟路 / 巡行 / 季节变体 / 传闻板 / 同伴合同 / 回流承接 / 资源家族` 等术语。
- 用户问题：每个词单独看都能理解，但堆在一页里时，玩家会先记住“系统很多”，而不是记住“怎么更顺手地玩”。
- 本轮建议：
  - 给高频术语补统一的一句话释义，首次出现时可见，后续弱化。
  - 将术语分层：首屏只出现结果型词汇，进阶展开再出现机制型词汇。
  - 优先把“勘明 / 熟域 / 回流承接”这种偏内部表达，调整成更玩家口径的说法。
- 预期效果：玩家更容易记住玩法节奏，不容易被名词劝退。

### 2.5 回城承接已经有了，但仍然更像“旅后说明”，还不是“回城办事单”

- 现状依据：`JourneySettlementReveal` 与页内旅后台账已经能展示旅程回顾、回流分发、后续去向、推荐动作，但仍需要玩家自己判断先后顺序。
- 用户问题：玩家知道“可以去很多地方”，却未必知道“应该先去哪个、现在去的收益最大”。
- 本轮建议：
  - 把旅后处理固定收成三步：先交差、再变现、后处理后续解锁。
  - 对每条建议动作补明确状态：`现在就去 / 可稍后去 / 本周可忽略`。
  - 页面内支持“标记已处理”，让办事单能逐步消掉，而不是一直悬在页上。
- 预期效果：回城后的体验从“读完一堆承接提示”变成“顺手把这一趟收尾做完”。

### 2.6 失败与撤退虽然不再静默，但恢复路径还不够“安抚式”

- 现状依据：当前已经有风险摘要、失败回滚路线和禁用原因，但失败后的恢复仍偏系统说明，情绪缓冲不够。
- 用户问题：玩家知道自己输了，但不一定知道“输了以后先干嘛最划算”，也不一定感到“这趟虽然输了，但世界仍然前进了”。
- 本轮建议：
  - 失败后固定给一条恢复建议：补给、低压路线、推荐承接页三选一。
  - 在失败摘要里突出“保住了什么 / 下次能少走什么弯路 / 新解锁了什么恢复机会”。
  - 对首领失败尤其要强调“下一次怎么更稳”，而不是只写失败结果。
- 预期效果：失败不只是挫败结算，而是能自然把玩家送回下一次尝试的起点。

### 2.7 历史、台账、回看信息开始出现重复，容易形成滚动疲劳

- 现状依据：当前页内同时存在“最近远征记录”“最新旅后分发”“选中旅后处理台账”“资源家族总览”“结算弹层”，内容有明显重叠。
- 用户问题：玩家会感觉信息很丰富，但也容易产生“好像哪里都能看，结果哪块都得翻”的疲劳。
- 本轮建议：
  - 页内常驻只保留“最近一次旅后处理”与“历史入口”。
  - 历史列表默认只展示最近 1-2 条，其余进入单独回看层。
  - 资源台账与旅后分发减少重复展示，避免同一批收益在一个页面出现三遍。
- 预期效果：信息仍然完整，但不会把主流程埋在回看内容里。

## 3. 移动端适配建议

### 3.1 优先压缩顶部全局信息层，让内容更早进入视口

- 现状依据：`GameLayout.vue` 在移动端仍保留 `StatusBar + 睡觉按钮 + TopGoalsPanel + 页面内容` 的纵向叠层；`docs/ui-smoke-2026-04-24/09-shop-mobile.png` 首屏上半段几乎都被全局信息占据。
- 建议动作：
  - 睡觉按钮并回状态栏操作区，避免独立占一整行。
  - `TopGoalsPanel` 移动端默认更薄，只保留“当前主线 + 一个 CTA + 可领提醒”。
  - 对手机首屏设明确目标：进入任意玩法页时，至少露出一个真正可操作的业务模块。
- 优先级：P0

### 3.2 合并右侧悬浮按钮，减少遮挡与误触

- 现状依据：`GameLayout.vue` 移动端右侧固定放了地图、设置、邮件、虚空箱、日志五个按钮；在 `09-shop-mobile.png` 中它们已开始侵占正文边缘。
- 建议动作：
  - 合并成一个主 FAB，点击后展开二级快捷操作，或改成底部工具抽屉。
  - 只把“当前最需要的 1-2 个动作”放在首层，例如地图与邮件。
  - 未读、可领奖之类的提醒保留，但不要让整列按钮长期常驻。
- 优先级：P0

### 3.3 行旅图移动端不能只靠“多折叠”，要改成任务流重排

- 现状依据：`RegionMapView.vue` 已经有 `mobilePrepExpanded / mobileHistoryExpanded / mobileLedgerExpanded / mobileSelectedAftermathExpanded` 等多个折叠开关，说明当前思路仍偏“把桌面信息折起来”。
- 建议动作：
  - 把移动端行旅图重排成 5 步：选区 -> 选路 -> 出发 -> 进行中 -> 回城处理。
  - 每一步只保留当前需要的 1 个主面板，其他信息放到“更多信息”里。
  - 对进行中远征和旅后处理，优先做整屏流程，而不是在长页面里继续堆区块。
- 优先级：P0

### 3.4 收口双轨交互，避免“横滑看地图、下翻点路线”的切换负担

- 现状依据：移动端区域内既有横向卷轴路网，又有下方路线清单；页面还直接提示“左右滑动看路段，路线按钮仍以下方清单为准”，说明当前存在两套并行交互。
- 建议动作：
  - 明确移动端主交互只保留一套：要么地图主导、点节点出路线卡；要么路线卡主导、地图只做情境预览。
  - 如果保留横向卷轴，需加当前节点、已看进度和可滑动提示，避免玩家看不到重点。
  - 地图和路线卡必须保持单向联动，不能让玩家在两个区域里重复做同一个选择。
- 优先级：P0

### 3.5 结算与旅后处理建议统一成全屏分步体验

- 现状依据：`JourneySettlementReveal.vue` 已经有阶段化 reveal，但在手机上仍要处理较长内容、动作卡和回执块；`RegionMapView.vue` 的移动端结算弹层接近整屏。
- 建议动作：
  - 强制分三步：旅程回顾 -> 回流分发 -> 推荐去向。
  - 每一步只做一件事，底部固定“下一步 / 前往推荐页 / 关闭并稍后处理”。
  - 记住玩家上次看到哪一步，避免中断后重头再看。
- 优先级：P1

### 3.6 提升可读性和点击热区，减少 10px 文本与小按钮堆叠

- 现状依据：`RegionMapView.vue`、`TopGoalsPanel.vue`、`MobileMapMenu.vue` 中大量使用 `text-[10px]`，移动端按钮也经常是紧凑式文字按钮。
- 建议动作：
  - 手机端正文与说明文字尽量抬到 12px 以上，关键数值与状态再做层级区分。
  - 所有主操作热区按 44px 及以上标准重审。
  - 减少一屏内的 chip 数量，优先保留结论型标签，弱化统计型标签。
- 优先级：P1

### 3.7 地图菜单需要更像“最近要去哪里”，而不是“全量后台入口表”

- 现状依据：`MobileMapMenu.vue` 目前按大类平铺很多入口，结构完整，但对手机用户来说更像目录，不像行动入口。
- 建议动作：
  - 在顶部增加“最近访问 / 当前推荐 / 可领奖 / 进行中”四类快捷目的地。
  - 保留全量地图，但下沉到第二屏或展开区。
  - 与 `TopGoals`、行旅图旅后办事单打通，让地图菜单直接承接推荐动作。
- 优先级：P1

### 3.8 长页面性能与信息加载顺序需要继续收口

- 现状依据：`RegionMapView.vue` 当前会在同一页里计算和展示大量摘要、历史、台账、承接卡；移动端又要额外处理多层展开状态。
- 建议动作：
  - 对折叠区做延迟渲染，避免首屏把所有模块都算一遍。
  - 历史、台账、传闻板等非首屏必要内容，在用户展开前不要完整挂载。
  - 移动端优先保证“进入页 -> 看懂主动作 -> 执行动作”的流畅度。
- 优先级：P1

## 4. 优先级排序

### A. 立刻要做的 5 件事（P0）

1. **把行旅图首屏收口成“本周建议动作”**
   - 理由：这是当前目标不清的根源。
   - 影响面：`RegionMapView`、`TopGoalsPanel`、`frontierDigest` 文案口径。

2. **压缩移动端顶部全局信息层**
   - 理由：现在很多页面首屏还没进入业务内容，已经先滚过一轮状态与规划。
   - 影响面：`GameLayout`、`StatusBar`、`TopGoalsPanel`。

3. **重组移动端悬浮按钮与地图菜单**
   - 理由：当前快捷入口太多、太常驻，既遮挡内容，也不利于单手操作。
   - 影响面：`GameLayout`、`MobileMapMenu`、全局导航。

4. **把旅后处理改成“回城办事单”**
   - 理由：玩法闭环已经有了，但执行闭环还差一步。
   - 影响面：`JourneySettlementReveal`、页内 aftermath、跨系统跳转。

5. **把路线卡改成更扫读的收益/风险预览**
   - 理由：玩家现在看到的是很多信息，不是很快的判断。
   - 影响面：`RegionMapView` 路线卡、区域卡、dispatch/handoff 摘要。

### B. 下一阶段可做的 5 件事（P1）

1. **统一术语分层与首次释义**
2. **减少历史 / 台账 / 回看内容重复**
3. **把移动端结算与旅后处理做成分步流程**
4. **收口卷轴路网与路线清单的双轨交互**
5. **对移动端折叠区做延迟渲染与性能收口**

### C. 长期统一规范 5 条（P2）

1. **所有中后期页面都要有“当前最推荐动作”单点出口**
2. **移动端首屏必须先露出一个真实可操作模块**
3. **解释型信息默认后置，结论型信息默认前置**
4. **跨系统承接统一改成“办事单 / 回执 / 下一步”结构**
5. **复杂玩法统一采用“流程分步”而不是“长页堆叠”**

注：本轮不建议继续优先扩更多区域内容、更多传闻数量或更多首领层数。当前更值钱的是把现有骨架变得更清楚、更顺手、更适合手机。

## 5. 建议验证方式

### 5.1 任务流走查

建议至少覆盖下面 4 条真实任务流：

1. **首次进入已解锁行旅图**
   - 验证目标：30 秒内能说出“本周建议去哪、为什么去、点哪里出发”。

2. **进行中远征中途继续**
   - 验证目标：玩家能在不读长说明的情况下，看懂当前节点、风险和下一步选择。

3. **回城后处理一趟收益**
   - 验证目标：玩家能在 2 次以内跳转里完成“先处理哪一项最值”的动作。

4. **失败或主动撤退后恢复**
   - 验证目标：玩家能立刻知道“先补什么、走哪条恢复线、何时再冲”。

### 5.2 移动端 smoke 场景

建议至少保留下面的移动端截图和交互验证：

- `390 x 844`：当前 smoke 基线。
- `360 x 780`：更窄安卓机型。
- `430 x 932`：更高机型，检查底部动作与安全区。
- 行旅图进行中远征页、旅后处理页、地图菜单页各做一轮单独截图。

当前工作区最新证据：

- 已补 `docs/ui-smoke-2026-04-26/summary.json` 与 12 张截图。
- 已覆盖 `390 x 844 / 360 x 780 / 430 x 932` 三档视口。
- 商店 / 任务 / 行旅图三页首屏都已露出主操作模块，且 summary 里未出现横向溢出、console error 或 page error。
- 同一份 summary 现还补进了 `钱袋 / 公会 / 鱼塘` 的 `390 x 844` 首屏截图，用于验证这套移动端三段式已经开始从高频页往更多中后期页扩展。

检查项：

- 首屏是否过度被状态栏和规划区占满。
- 右侧或底部快捷入口是否遮挡正文。
- 是否存在需要横向滑动但无明显提示的区域。
- 全屏弹层关闭、返回、继续操作是否顺手。

### 5.3 可用性验收阈值

本轮建议给几个可以直接执行的验收阈值：

- 手机首屏进入玩法页后，**至少露出 1 个主操作模块**。
- 行旅图从进入到发起一次远征，**理想不超过 3 次主点击**。
- 回城后从看到结算到进入首个推荐承接页，**理想不超过 2 次主点击**。
- 主要操作按钮点击热区 **不低于 44px**。
- 主要说明文案在手机上 **尽量不低于 12px**。

当前这轮已验证通过的阈值：

- 商店 / 任务 / 行旅图在三档手机视口首屏都能直接露出 1 个主操作模块。
- 地图菜单已收口为 1 个常驻主入口，再进入二级工具层。
- 行旅图移动端横向卷轴已改成只做横向对齐，不再为了聚焦节点把“当前建议动作”自动滚出首屏。

### 5.4 上线后观察项

如果后续补埋点，建议重点看：

- 行旅图进入后首个点击是否集中到推荐路线。
- 旅后处理后的跨系统跳转是否更集中、是否更快发生。
- 失败后是否能更快回到恢复路线，而不是直接退出行旅图。
- 移动端页面停留时长是否下降，但有效操作率是否上升。

## 6. 一句话结论

`0425plan` 解决的是“行旅图要长成什么样”，`0426plan` 更应该解决“玩家现在能不能顺手玩明白”。

本轮最值得做的，不是继续加内容，而是把已经做出来的内容收成一条清楚的行动路径，尤其要先把手机上的首屏压力、操作分散和回城承接说明化问题收掉。
